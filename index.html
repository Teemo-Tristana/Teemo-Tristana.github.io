Go 程序结构

命名

  1. Go 语言中的函数名、变量名、常量名、类型名、语句标号和包名等所有的命名，都遵循一个简单的命名规则 : 以字母（Unicode字母）或下划线开头。

  2. Go 语言的 25 个关键字(关键字不能用于自定义名字，只能在特定的程序结构中使用)

  3. Go语言中，有 30 多个预定义的名字，主要是内置常量、类型和函数

    内部预定义的名字不是关键字,因此可以重新使用它们，但是尽量少重复使用，避免引起语义混乱。

  4. 函数名字的开头字母的大小写决定了该函数在包外的可见性，若是大写字母开头，则是可导出的，比如 fmt的Println()函数

  5. 推荐使用 驼峰式 命名

声明

Go 语言主要有四种类型的声明语句：

  • var 用于 变量

  • const 用于 常量

  • type 用于 类型

  • func 用于 函数

    变量

    包一级的声明可以对整个包有效，而函数内部的声明只能局部有效

  • 变量的声明方式

    1. var 变量名 类型 = 表达式
      如果初始化表达式被省略，那么将用零值初始化该变量。 数值类型变量对应的零值是0，布尔类型变量对应的零值是false，字符串类型对应的零值是空字符串，接口或引用类型（包括slice、指针、map、chan和函数）变量对应的零值是nil。数组或结构体等聚合类型对应的零值是每个元素或字段都是对应该类型的零值。
      零值初始化机制可以确保每个声明的变量总是有一个值，因此 Go 语言中不存在未初始化的变量。

    2. 多种类型同时声明

    3. 简短变量声明
      名字 := 表达式

指针

一个指针的值是另一个变量的地址。一个指针对应变量在内存中的存储位置。并不是每一个值都会有一个内存地址，但是对于每一个变量必然有对应的内存地址

任何类型的指针的零值都是 nil

Go 语言中返回函数局部变量的地址也是安全的

new函数

new 函数开业用于创建变量， new(T) 将创建一个T类型的匿名变量，并初始化为T类型的零值,并返回 变量地址(即指针), 返回类型是 T 类型的指针(T*)

用new创建变量和普通变量声明语句方式创建变量没有什么区别，换言之，new函数类似是一种语法糖，而不是一个新的基础概念。

变量的声明周期

变量的生命周期指的是在程序运行期间变量有效存在的时间段。对于在包一级声明的变量来说，它们的生命周期和整个程序的运行周期是一致的。而相比之下，局部变量的生命周期则是动态的：每次从创建一个新变量的声明语句开始，直到该变量不再被引用为止，然后变量的存储空间可能被回收。函数的参数变量和返回值变量都是局部变量。它们在函数每次被调用的时候创建。

Go 语言的自动垃圾收集器(基本思路)：
从每个包级的变量和每个当前运行函数的每一个局部变量开始，通过指针或引用的访问路径遍历，是否可以找到该变量。如果不存在这样的访问路径，那么说明该变量是不可达的，也就是说它是否存在并不会影响程序后续的计算结果

编译器会自动选择在堆或栈上分配局部变量的存储空间，但是，并不是由 var 或 new 声明方式决定的，有编译器决定。

逃逸的变量需要额外分配内存，同时对性能的优化可能会产生细微的影响。

类型

变量或表达式的类型定义了对应存储值的属性特征

type 类型名字 底层类型

包 和 文件

Go 语言中的 包 类似其他语言的模块或库，目的是为了支持模块化、封装、单独编译和代码复用。

数据类型

Go 语言将数据类型分为四类 : 基础类型、复合类型、引用类型和接口类型

基础类型

基础类型比如 数字、字符串、布尔等类型

  • 整数类型

    • Go 提供了四种不同大小的整数类型: int8/int16/int32/int64 和 uint8/uint16/uint32/uint64。

    • 也提供了特定的CPU的 int 和 uint(32bit/64bit),具体大小与编译器有关。

    • Go采用的是 Unicode 字符，UTF8 编码， 它的 rune 类型和 int32 等价，表示一个 Unicode 码点

    • byte 和 uint8 等价，一般用于数值。

    • 不管它们的具体大小，int、uint和uintptr是不同类型的兄弟类型。其中int和int32也是不同的类型，即使int的大小也是32bit，在需要将int当作int32类型的地方需要一个显式的类型转换操作。

  • 浮点数类型

    • Go语言提供了两种精度的浮点数，float32和float64

  • 复数
    Go语言提供了两种精度的复数类型：complex64和complex128，分别对应float32和float64两种浮点数精度。内置的complex函数用于构建复数，内建的real()和imag()函数分别返回复数的实部和虚部

  • 布尔类型

    • 一个布尔类型的值只有两种：true和false。if和for语句的条件部分都是布尔类型的值，并且==和<等比较操作也会产生布尔型的值。

  • 字符串

    • 一个字符串是一个不可改变的字节序列(字符串是只读)。字符串可以包含任意的数据，包括byte值0，但是通常是用来包含人类可读的文本。文本字符串通常被解释为采用UTF8编码的Unicode码点（rune）序列。
    • len() 返回字符串的字节数目(而不是 rune 字符数目)
    • 标准库中有四个包对字符串处理尤为重要：bytes、strings、strconv和unicode包
      • strings 提供字符串相关操作比如替换、查询、比较等
      • bytes 功能和 strings 包类型，但是只要是争对 []byte 类型
      • strconv包提供了布尔型、整型数、浮点数和对应字符串的相互转换，还提供了双引号转义相关的转换。
      • unicode包提供了IsDigit、IsLetter、IsUpper和IsLower等类似功能，它们用于给字符分类

  • 常量

    • 常量表达式的值在编译期就已经确定了，所有常量的运算都可以在编译期完成，这样可以减少运行时的工作，也方便其他编译优化
    • iota
      • 在一个const声明语句中，在第一个声明的常量所在的行，iota将会被置为0，然后在每一个有常量声明的行加一。

初识数据库

DBMS的种类

数据库是将大量数据保存起来，通过计算机加工而成的可以进行高效访问的数据集合。该数据集合称为数据库（Database，DB）。用来管理数据库的计算机系统称为数据库管理系统（Database Management System，DBMS）。

DBMS 主要通过数据的保存格式（数据库的种类）来进行分类，现阶段主要有以下 5 种类型.

  • Oracle Database：甲骨文公司的RDBMS
  • SQL Server：微软公司的RDBMS
  • DB2：IBM公司的RDBMS
  • PostgreSQL：开源的RDBMS
  • MySQL：开源的RDBMS

常见DBMS系统结构图

使用 RDBMS 时，最常见的系统结构就是客户端 / 服务器类型（C/S类型）这种结构

SQL

数据库中存储的表的结构类似 excel 中的行和列。在数据库中，行称为记录，也即是一条数据的记录。列称为字段，代表了表中存储的数据项目。行和列的交汇位置称为单元格。

SQL指令

SQL的书写顺序:

select -> from -> where -> group by -> having -> order by

SQL的执行顺序:

from -> where -> group by -> having -> select -> order by

根据对 DBMS 赋予的指令的不同，SQL语句可以分为四类：

  • DDL(Data Definition Language:数据定义语言)：用于创建或删除存储数据用的数据库或表等对象，DDL包含了以下指令：
    • create ： 创建数据库/表等对象
    • drop ： 删除数据库/表等对象
    • alter ： 修改数据库/表等对象。
  • DML(Data Manipulation Language:数据操作语言)：用于查询或变更表中的记录，DML 包含了以下指令：
    • select ：查询表中数据
    • insert ：向表中插入数据
    • update ：更新表中数据
    • delete ：删除表中数据
  • DCL(Data Control Language: 数据控制语言)：用于确认或取消对数据库中的数据的变更以及权限设置等。DCL 包含了以下指令：
    • commit : 确认对数据库中数据的变更
    • rollback : 取消对数据中的数据进行的变更(回滚)
    • grant : 赋予用户权限
    • revoke : 取消用户的操作权限

数据类型

数据库创建的表，所以列都必须指定数据类型，且每列存储的数据类型必须相符。

  • interger ： 用来指定存储整数的列(数字类型)，不能用于存储小数

  • char ： 用来存储指定长度的字符串，若长度不足则用空格填充，超过则会截取[由于会浪费存储空间，一般很少用]

  • varchar ： 用于存储可变长度的字符串.

  • date ： 用来存储日期的类型

约束

约束是除了数据类型之外，对列中的数据进行的限制(追加)条件

NOT NULL 表示非空约束, 该列必须有数据输入

primary key 表示主键约束, 该列是唯一值，可以通过该列取出特定的行的数。

表的删除与更新

删除

drop、delete、truncate

  1. 一般来说，三者删除速度：drop>truncate>delete；

  2. delete是DML，需要手动提交操作才能生效，可以回滚，可以触发触发器；truncate和drop是DDL，会隐式提交，不能回滚，不会触发触发器。

    • 执行delete操作时，每次从表中删除一行，并且同时将该行的的删除操作记录日志文件中。delete可根据条件删除表中满足条件的数据，如果不指定where子句，则跟truncate一样，可以删除表中所有记录。

    • truncate会一次性删除表中所有记录，通过释放存储表数据所用的页来删除数据，不会将删除的数据记录在日志文件中，只在日志文件中记录页的释放，使用的系统和事务日志资源少，所以执行速度比delete快。对于外键（foreignkey ）约束引用的表，不能使用 truncate table，而应使用不带 where 子句的 delete 语句。

    • drop则删除整个表的结构和数据，但是被依赖的约束、触发器、索引、依赖于该表的存储过程、函数将被保留，但其状态会变为invalid。

  4. 删除后，表和索引所占空间：

    • delete操作不会减少表或索引所占用的空间；
    • 当表被truncate后，这个表和索引所占用的空间会恢复到初始大小，但表结构及其列、约束、索引的定义等保持不变。
    • drop语句将表所占用的空间全释放掉。

  5. 使用场景：

  • 如果想删除表，当然用drop；

  • 如果想保留表而将所有数据删除，而且和事务无关，用truncate即可；

  • 果和事务有关，或者想触发trigger，还是用delete；

  • 删除表：

    drop table <表名>
    drop table product; //删除 produtc 表
    Note: drop删除的表是无法恢复的，只能通过新插入恢复。

  • 给已有的表新增加列

    alter table <表名> add column <列的定义>
    alter table product add column product_name_pingyin varchar(100);// 添加一列可以存储100位的可变长字符串的 product_name_pinyin 列

  • 删除列

    alter table <表名> drop column <列名>;
    alter table product drop column product_pingyin;//删除 product_name_pinyin 列(删除后无法恢复)

  • 清空表的内容

    truncate table <表名>\

    truncate table product; // 清空product表\
相比drop / delete，truncate用来清除数据时，速度最快。(可恢复？)

对表的聚合查询

聚合函数
5个聚合函数

count(): 计算表中记录数目(行数)
sum(): 计算表中数值列的总和
avg(): 计算表中数值的平均值
max(): 计算指定列的最大值
min(): 计算指定列的最小值

  • COUNT函数的结果根据参数的不同而不同。COUNT(*)会得到包含NULL的数据行数，而COUNT(<列名>)会得到NULL之外的数据行数。
  • 聚合函数会将NULL排除在外。但COUNT(*)例外，并不会排除NULL。
  • MAX/MIN函数几乎适用于所有数据类型的列。SUM/AVG函数只适用于数值类型的列。
  • 想要计算值的种类时，可以在COUNT函数的参数中使用DISTINCT。
  • 在聚合函数的参数中使用DISTINCT，可以删除重复数据。

对表进行分组

group by

  • 在 GROUP BY 子句中指定的列称为聚合键或分组列。

  • 顺序：

    select -> from -> where -> group by

  • group by 子句不能使用 select 子句定义的别名，但是在 order by 中却可以使用。

原因: 在 SQL 中使用 having 子句时，select 语句的顺序为:
from -> where -> group by -> having -> select -> order by
其中 select 的执行顺序在 group by 之后，在 order by 之前，也就是说，当在ORDER BY中使用别名时，已经知道了SELECT设置的别名存在，但是在GROUP BY中使用别名时还不知道别名的存在，所以不能在ORDER BY中可以使用别名，但是在GROUP BY中不能使用别名

having

having 子句用于对分组进行过滤，可以使用数字、聚合数和group by 中指定的列名(聚合键)

order by

SQL 执行的结果是随机排序的，当需要指定顺序排序时，可使用 order by 子句进行排序。

  • SQL 默认是升序排列(asc), 降序排列则为(desc)

视图

表面上来看视图和表是相同的，两者大部分操作相同。那为什么会有视图呢？

什么是视图

视图是一张虚拟的表, 不同于实际存在的表，视图是由 select 语句创建而来，因此，视图是由 select 语句生成的一张虚拟的表，操作视图即操作这张虚拟的表[零时缓存?]

视图与表

视图与表的区别:是否保存了实际的数据，视图和数据表的本质，即视图是基于真实表的一张虚拟的表，其数据来源均建立在真实表的基础上。 视图并不是真正存储数据，它是一个窗口，通过这个窗口可以看到表中的真实数据。

“视图不是表，视图是虚表，视图依赖于表”

视图存在的原因：

  • 通过视图可以将频繁使用的select 语句保存以提高效率。
  • 通过视图可以使用户看到的数据更加清晰(?)
  • 通过视图可以不对外公开表的全部字段，增强数据的保密性。
  • 通过视图可以降低数据的冗余

创建视图

视图中保存的是 select语句。

CREATE VIEW <视图名称>(<列名1>,<列名2>,…) AS <SELECT语句> //SELECT 语句中列的排列顺序和视图中列的排列顺序相同

视图名称在数据中必须唯一，不能与其他视图或表重名，视图可以基于表，也可以基于其他视图。但是多重视图会降低 SQL的性能，因此，一般少用。

视图操作

因为视图是一个虚拟表，所以对视图的操作就是对底层基础表的操作，所以在修改时只有满足底层基本表的定义才能成功修改。

视图本质上还是由表派生而来，若原表更新，则视图也会被更新，反之亦然(修改视图的数据，原表的数据也会更新，否则两者的数据会不一致)。

视图只是原表的一个窗口，因此它只能修改通过窗口能够看到的内容(数据)[视图是一个窗口，类似临时缓存？]

不推荐通过视图修改原表数据。

子查询

子查询是指一个查询语句嵌套在另外一个查询语句内部的查询。在 select 子句中线计算子查询，子查询结果作为外层查询的过滤条件。

子查询与视图

子查询就是将用来定义视图的 select 语句直接作为子句。但子句查询是一次
性的，而视图则可以保存在存储介质中，后续可以继续用。

嵌套子查询

嵌套子句可以查询出结果，但是随着嵌套子句层数的增加，SQL语句难以理解同时会 执行效率。

标量子查询

标量就是单一的意思，那么标量子查询也就是单一的子查询，那什么叫做单一的子查询呢？

所谓单一就是要求我们执行的SQL语句只能返回一个值，也就是要返回表中具体的某一行的某一列

关联子查询

关联子查询包含了关联的两字段，因此，意味着查询与查询之间存在某种关联。

关联子查询就是通过一些标志将内外层的查询连接起来起到过滤数据的目的

  • 关联子查询的执行逻辑完全不同于正常的SELECT语句。

    • 正常的select 语句(或者说简单的select语句)按照

      from -> where -> group by -> having -> select -> order by

      的来执行。

    • 关联子查询，则是从先执行主查询，然后从主查询结果中逐步去执行子查询。

      1. 首先执行不带 where 的主查询
      2. 根据主查询结果匹配 指定条件， 获取子查询结果
      3. 将子查询结果与主查询结果执行完整的 SQL 语句。
        具体可参考连接

    • 常规回答
      • 进程是资源分配的最小单位，线程是CPU调度的最小单位
    • 深入回答
      • 从 Linux 内核实现出发， Linux 中进程与线程是如何创建的，区别在哪儿？
        • Linux 中，进程和线程实际上都是用 task_struct 结构体来执行一个进程实体；但是进程创建使用的是 fork() 系统调用，而线程创建则用的是 pthread_create() , 这两种方法在内部都是调用 do_fork() 来执行创建操作，但是它会根据传入的参数不同执行不同的操作。
        • Linux实现线程的方式很巧妙，从使用者角度来讲，有线程与进程之分，但是实际上根本没有线程：pthread_create()调用do_fork()时，创建的仍然是进程，但是通过参数指定了多个进程(线程)之间共享某系资源(比如虚拟内存、页表、文件描述符等)，函数调用font color=”#00dddd”>栈、寄存器等则为线程私有的数据。

RPC

发表于 2020-11-20

RPC

相关概念

什么叫RPC？

RPC全称是Remote Procedure Call，即远程服务调用:两台服务器A、B，一个部署在A服务器上，
想要调用B服务器上的提供的函数/方法，由于不在不同的服务器上，需要通过网络来传数据
从而实现调用。
RPC不是一种协议，而是一种概念或方法。RPC就是要像调用本地函数一样去调用远程函数

为什么要用RPC？

RPC的主要目标是让构建的分布式计算应用更容易、更透明，让使用者不必显式的区分
本地调用和远程调用。

RPC(远程过程调用)带来的问题？

在远程调用时，需要执行的函数体是在远程的机器上的，这几带来几个问题：

  1. Call ID 映射
    怎么告知远程机器(服务端)哪一个才是需要被调用的函数(寻址问题)?

在本地调用中，函数体是通过函数指针来指定的，调用某个函数比如add()，是
直接通过函数指针来指定的，具体由编译器实现，编译器自动调用相应的函数指针。

远程调用时，是在不同的机器上，不同直接使用函数指针,因此在不同的机器上上，
两个进程的地址空间是有可能不一样的。
因此，在 RPC 中，每个函数拥有一个ID，这个ID在所有进程中是唯一的，
在远程过程调用时，附带上这个ID。同时，在客户端和服务端分别维护一个 {函数 <–> Call ID}
对应表，两者的表可以不同，但是相同的函数对应的ID必须时相同的。
当客户端需要进行远程调用时，查询该表，获取相应函数的ID，然后将其传给服务端
服务端通过ID查询该表，来确定客户端需要调用的函数，然后执行相应的代码。

  1. 序列化与反序列化(XML/JSON/二进制)
    客户端怎么把参数值传给远程的函数?

在本地调用中，通常是将参数压栈，然后被函数进行读取(弹栈)即可(本地函数调用
时，不一定使用栈，编译器可以进行优化，比如使用寄存器或将函数进行内联(inline))。

远程过程调用时，客户端和服务端时不同的进程，是不能通过内存来传递参数的，
甚至客户端和服务端使用都不是同一种编程语言。
因此，通常采用的方法是将客户端的参数转为字节流(序列化)，然后传递给服务端后，
再把字节流转为它能读取的格式(反序列化)，也即是序列化和反序列化。同理，
服务端的返回值也会有序列化与反序列化的过程。

  1. 网络传输
    RPC(远程过程调用)通常通过网络将客户端和服务端连起来，两者之间的数据需要通过网络
    参数；因此，就会需要用到一个网络传输层。

网络传输层的功能是将Call ID 和序列化后的参数的通过网络层传给服务端，服务端执行后
在将结果序列化并传给客户端。只要能实现这个功能的，就可以作为传输层使用。

大部分RPC框架使用的是TCP协议，但UDP也是可以的，google的gRPC使用的是HTTP2.0。
