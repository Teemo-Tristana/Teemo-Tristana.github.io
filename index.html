Redis事件

发表于 2020-11-24

Redis事件

在 Redis 中， 事件分为两大类：文件事件和时间时间。
文件事件指代的是主要包含以下三类：
客户端的网络连接请求达到事件
客户端发来的命令请求达到事件
服务端发出命令应答事件

文件事件（file event）

Redis 服务器通过套接字与客户端进行连接，文件事件就是服务器对套接字操作的抽象
服务器与客户端的通信会产生相应的文件事件，而服务器则通过监听并处理这些事件来完成一些列网络操作。

文件事件的处理器

Redis为文件事件编写了多个处理器，用于处理不同的网路请求

  1. 连接应答处理器 , 用于处理客户端的应答
    • newworking.c/acceptTcpHandler(),用于对连接服务器监听套接字的客户端进行应答
    • Redis 服务器初始化时，会将 连接应答处理器和服务器监听套接字的 ae_readable 事件关联起来，当有 connect 连接事件时，会触发 ae_readable，从而引发连接应答处理器执行相应的操作。
  2. 命令请求处理器 , 用于处理客户端的请求
    • newworking.c/readyQueryFromClient(), 从套接字中读取客户端发送的请求命令
    • 在客户端与服务器连接连接后，客户端发送请求命令给服务器时，套接字会产生 readable 事件， 从而引发执行命令请求处理器
    • 在客户端连接服务器期间，此关联一直存在
  3. 命令回复处理器 , 用于将服务器处理结果返回给客户端
    • newworking.c/sendReplyToClient(), 将服务器处理结果通过套接字返回给客户端
    • 服务器有命令回复给client时，会将套接字的 ae_writable 与 命令恢复执行器关联，当有命令回复时，会产生 ae_writable，从而引发回复命令执行器的执行
    • 执行完毕后，服务器就会解除此关联
  4. 复制处理器 , 用于主从服务器的复制操作
    其中，前三个最常使用。

    时间事件(time event)

    Redis 服务器的一些操作需要定期执行，时间事件就服务器对此类定时操作的抽象

时间事件主要是 Redis 内部的定时处理器

RPC

发表于 2020-11-20

RPC

相关概念

什么叫RPC？

RPC全称是Remote Procedure Call，即远程服务调用:两台服务器A、B，一个部署在A服务器上，
想要调用B服务器上的提供的函数/方法，由于不在不同的服务器上，需要通过网络来传数据
从而实现调用。
RPC不是一种协议，而是一种概念或方法。RPC就是要像调用本地函数一样去调用远程函数

为什么要用RPC？

RPC的主要目标是让构建的分布式计算应用更容易、更透明，让使用者不必显式的区分
本地调用和远程调用。

RPC(远程过程调用)带来的问题？

在远程调用时，需要执行的函数体是在远程的机器上的，这几带来几个问题：

  1. Call ID 映射
    怎么告知远程机器(服务端)哪一个才是需要被调用的函数(寻址问题)?

在本地调用中，函数体是通过函数指针来指定的，调用某个函数比如add()，是
直接通过函数指针来指定的，具体由编译器实现，编译器自动调用相应的函数指针。

远程调用时，是在不同的机器上，不同直接使用函数指针,因此在不同的机器上上，
两个进程的地址空间是有可能不一样的。
因此，在 RPC 中，每个函数拥有一个ID，这个ID在所有进程中是唯一的，
在远程过程调用时，附带上这个ID。同时，在客户端和服务端分别维护一个 {函数 <–> Call ID}
对应表，两者的表可以不同，但是相同的函数对应的ID必须时相同的。
当客户端需要进行远程调用时，查询该表，获取相应函数的ID，然后将其传给服务端
服务端通过ID查询该表，来确定客户端需要调用的函数，然后执行相应的代码。

  1. 序列化与反序列化(XML/JSON/二进制)
    客户端怎么把参数值传给远程的函数?

在本地调用中，通常是将参数压栈，然后被函数进行读取(弹栈)即可(本地函数调用
时，不一定使用栈，编译器可以进行优化，比如使用寄存器或将函数进行内联(inline))。

远程过程调用时，客户端和服务端时不同的进程，是不能通过内存来传递参数的，
甚至客户端和服务端使用都不是同一种编程语言。
因此，通常采用的方法是将客户端的参数转为字节流(序列化)，然后传递给服务端后，
再把字节流转为它能读取的格式(反序列化)，也即是序列化和反序列化。同理，
服务端的返回值也会有序列化与反序列化的过程。

  1. 网络传输
    RPC(远程过程调用)通常通过网络将客户端和服务端连起来，两者之间的数据需要通过网络
    参数；因此，就会需要用到一个网络传输层。

网络传输层的功能是将Call ID 和序列化后的参数的通过网络层传给服务端，服务端执行后
在将结果序列化并传给客户端。只要能实现这个功能的，就可以作为传输层使用。

大部分RPC框架使用的是TCP协议，但UDP也是可以的，google的gRPC使用的是HTTP2.0。

Redis持久化

  • Redis是内存数据库，其数据保存在内存中，若不保存到磁盘，则一旦服务器宕机，则数据会丢失
  • 为了让这些数据在 Redis 重启之后仍然可用， Redis 分别提供了 RDB 和 AOF 两种持久化模式。

如何阅读Redis源码

    1. 阅读数据结构的实现：
      • 相关文件：
        • sds.h sds.c
        • adlist.h adlist.c
        • dict.h dict.c
        • redis.h zskiplist zskiplistNode t_zset.c 所有zsl开头的函数如zslCreate, zslInsert, zslDeleteNode
        • hyperloglog.c 中hlhdr结构，以及所有hll开头的文件
    1. 阅读内存编码[encoding]数据结构的实现
      • 和普通的数据结构一样， 内存编码数据结构基本上是独立的， 不和其他模块耦合
        • intset.h intset.c[整数集合intset数据结构]
        • ziplist.h ziplist.c[压缩列表zip-list数据结构]
    1. 阅读数据类型实现：
      • Redis6种不同类型的键[字符串-散列-列表-集合-有序集合-HyperLogLog]所有底层实现结构
        • object.c Redis对象类型系统实现
        • t_string.c 字符串键的实现
        • t_list.c 列表键的实现
        • t_hash.c 散列键的实现
        • t_set.c 集合键的实现
        • t_zset.c中除zsl开头的函数之外的所有函数 有序集合键的实现
        • hyperloglog.c 中所有以pf开头的函数

    1. 阅读数据库相关实现代码:

      • 数据库相关

        • redis.h中redisDb结构和db.c Redis数据库实现
        • notify.c Redis的数据库通知功能实现代码
        • rdb.h和rdb.c Redis的RDB持久化实现代码
        • aof.c Redis的AOF持久化实现代码

      • Redis独立功能模块：

        • redis.h中pubsubPattern结构以及pubsub.c 发布与阅读功能实现
        • redis.h中multiState结构以及multiCmd结构 事务功能的实现
        • sort.c SORT命令的实现
        • bitops.c GETBIT|SETBIT等二进制操作命令的实现

    1. 阅读客户端和服务器相关代码：

      • 客户端-服务器代码：

        • ae.c 以及任意一个ae_*.c文件 Redis的事件处理器实现[基于Reactor模式]
        • networking.c Redis的网络链接库，负责发送回复和接受命令请求，同时也负责创建/销毁客户端以及分析通信协议等工作
        • redis.h 和 redis.c中和单机Redis服务器相关部分 单机Redis服务器的实现

      • 独立模块:

        • scripting.c Lua脚本功能实现
        • slowlog.c 慢查询功能的实现
        • monitor.c 监视器功能的实现
    1. 阅读多机功能的实现
      • 多级功能代码：
        • replication.c 复制功能的实现
        • sentinel.c Redis Sentinel的实现代码
        • cluster.c Redis集群的实现代码

什么是热key ？ 热key的问题怎么解决

  • 所谓 热key问题就是，突然有大量的请求比如几十万去访问redis某个特定的key,那么这样会导致流量过于集中，而到达物理网卡的上限，从而导致这台redis服务器宕机引发雪崩。
  • 针对热key的解决方案:
    1. 提前把热key打散到不同的服务器,降低压力
    2. 加入二级缓存，提前加载热key数据到内存中，若redis宕机，走内存查询。
