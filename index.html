解密Redis持久化

发表于 2020-11-13 更新于 2020-11-14

解密Redis持久化

  1. Redis作者关于Redis持久化的博文，http://oldblog.antirez.com/post/
    看博文、刷论坛以及在推特上搜索有关Redis的内容其实也是我开发Redis工作的一部分。对开发者来说，关注社区用户和非社区用户对你正在开发的产品看法是至关重要的。从自身感受来说，在Redis的所有特性里面，Redis的持久化是被误解最深。

在本篇博客中，我将尽可能公正、不偏不倚地阐述Redis的持久化：不吹不黑，不跳过可能导致陷入Redis困境的细节.尽量做到以清晰易懂的方式来阐述Redis的持久化工作原理,可靠性以及与其他数据库的对比。

os和磁盘

首先，考虑为什么要做持久化也即是数据库持久化的作用是什么？为此，先来看一次简单的写操作期间发生了什么。

    1. 客户端向数据库发送写命令(数据在客户端内存)
    1. 数据库收到写命令(数据在服务器内存)
    1. 数据库调用系统将数据写入磁盘(数据位于内核缓冲区中)
    1. 操作系统(内核)将写缓冲区数据传输到磁盘控制器(数据位于磁盘缓冲区)
    1. 磁盘控制器执行真正地写操作，将数据写入物理介质(磁盘、闪存)

Note: 以上步骤是已经简化后的过程，实际可能涉及到更多级缓存和缓冲

第2步通常是在数据库中实现为一个复杂地系统，写操作有时被不同地进程或线程处理。但是数据将在很快地时间内被写入磁盘，在此不做过多地累述。一句话，来自内存的数据将将在某一时刻传给内核(第三步)

在第3步中也有大量地省略。现代内存实现了不同等级的缓存，它们通常是文件系统级别的缓存(Linux称位页缓存)和较小的缓冲区，这些缓存区含有待提交到磁盘的数据。调用特殊的API可以跳过这两种缓存机制(比如open系统调函数的O_DIRECT和O_SYNC标识)。但是我们可以将这些缓存视为一层(即不关心其细节)。因此，我们认为如果数据库已实现了缓存机制来避免数据库和内核并行执行同一项工作，那么就可以关闭缓存页。缓冲区通常被保留下来，因为若关闭了缓冲区，每次写操作都使得写磁盘数据很慢(系统调用比较耗时)

数据库通常的做法是，只有在必要的时候才会调用系统调用将缓冲的数据写入磁盘，我们将在后面详细地介绍。

什么时候才是安全的？

如果考虑到数据库故障比如宕机(比如终止或崩溃)而内核没出问题，只有第3步成功执行后才会认为是安全的，也即是write(2)(或将数据传给内核的其他系统调用)成功返回。在完成此步骤后，即使数据库进程崩溃，内核仍然会将数据传给磁盘控制器。

考虑更为严重的情况，比如断电等，这样的话则只能确保第5步完成了才是安全的，第五步实际上是将数据保存到物理存储设备。

总结一下，数据安全的关键步骤是第3、4和5步，这几个步骤涉及到：

  • 数据库以什么频率的调用write(或等效的系统调用)将数据从用户空间传输到内核缓冲区？
  • 内核缓冲区将多久刷新一次，将数据从缓冲区flush到磁盘控制器？
  • 磁盘控制器在什么时候将数据写入到物理设备？

Note：所谓的磁盘控制器实际上指代的时驱动程序或磁盘自身的缓存行为，在持久化比较铭感(重要)的系统中，系统管理员通常禁用这种缓存方式。

默认情况下，大多数系统的驱动通常只执行直写式高速缓存(即，只高速缓存读，不高速写)。只有在提供足够电池(应对断电等)情况下，才会启动回写模式write_back_mode，以保证数据安全。

POSIX API

从数据库开发者角度来看，数据被真正写入物理设备(磁盘)前的流动方向是很有意思的，但更有意思的是控制这一流动方向的API。

分析第3步的过程，我们可以使用write系统调用将数据写入内核缓冲区，从这个角度来看，可以使用POSIX API来控制这一过程。但是我们并不知道该系统调用成功返回时需要的执行时间。内核写缓冲区大小是有限的，如果内核缓冲区无法满足写请求，导致缓冲区溢出，那么内核会阻塞该写操作，当内核可以接收更多数据时(即内核有空闲空间)时，write系统调用才会返回。总之，最终的目的是将数据写入物理介质。

同理第4步，在此步骤中，是将数据从内核缓冲区传输到磁盘控制器。默认情况下该步频率不是太频繁，因为传输越大的数据库时速度越快，比如，Linux默认30秒提交一次写操作。这意味着在提交之前这段时间，若发生故障，则最近30秒内待写入的数据就可能丢失。

POSIX API 提供一些列系统调用用来强制内核将缓冲区的数据写入磁盘：最著名的可能就是 fsync 系统调用(有关更多信息，另请参见 mysnc 和 fdatasync)。使用 fsync ，数据库系统可以强制将内核缓冲区数据真正地写入磁盘，不过你也应该可以想到，该该操作开销很大：fsync 每次调用时都会启动一次写操作并且内核缓存会遗留一些待处理的数据。 sync()会一直阻塞该进程直到写操作完成，如果有需要的话，在Linux系统下，它也会阻塞向同一个文件写入的所有其他线程。

控制之外的事情

到现在位置，第3、4步是可以控制的，那么第5步呢？严格的来说，无法使用 POSIX API 来控制第5步。有些内核实现或许会告知使用者驱动程序将数据真正提交给物理介质，又或许控制器为了将数据提高速度而将写操作进行重排，而非立即将数据写入磁盘，与此相反的是，可能需要等待若干毫秒。这些是我们无法控制和干预的。

在本文余下的部分，将简单地探讨简化后地两种数据安全级别：

  • 使用 write(2) (或等效)的系统调用将数据写入内核缓冲区，从而保证数据安全，避免进程受异常情况的影响
  • 使用 fsync(2) (或等效)的系统调用将数据从内核缓冲区写入到磁盘，这一个过程保证数据安全避免受到系统故障如断电的影响。但是，这一点是无法真正保证的，因为磁盘控制器可能存在缓存，但是可以不去考虑这情况，因为这种缓存在常用的数据库系统中是相同的。而且，数据库管理员可以通过特定的工具来管理物理设备的具体行为。

Note：并非所有的数据库都是用 POSIXAPI 。一些专用数据库使用内核模块，这些内核模块可以直接与硬件进行交互。但是该问题的本质是一样的。可以使用用户缓冲区、内核缓冲区，但迟早都需要将数据写入磁盘以保证其数据安全(这是一个耗时的过程)。Oracle就是一个使用内核模块的数据库实际例子。

Redis持久化

  • Redis是内存数据库，其数据保存在内存中，若不保存到磁盘，则一旦服务器宕机，则数据会丢失

  • 为了让这些数据在 Redis 重启之后仍然可用， Redis 分别提供了 RDB 和 AOF 两种持久化模式。

    RDB持久化：内存快照[Redis DataBase]

  • RDB的功能是记录Redis数据库在某一时刻状态的文件，该文件是一个经过压缩的二级制文件[类似照片记录下莫一个瞬间，相当于Linux下的ps命令]。该文件可以还原生成RDB文件时的数据库状态。

    RDB作用：

  • 在Redis运行时，RDB程序将当前Redis数据库的内存快照保存到磁盘中，再次重启时，RDB程序可以根据RDB文件来还原数据库状态

  • Redis的RDB是将数据库所有数都保存到RDB文件中，因此，RDB执行的是全量快照:将内存中的数据都保存到磁盘，好处是一次性记录所有数据，保证完整性，不足在于将数据库全部保存，开销大

    RDB两个命令：

  • 生成RDB文件的两个Reds命令：

    • save
      • save命令是在主线程中执行，因而会阻塞[阻塞Redis服务器进程]直到RDB文件完成
      • 执行 save 命令期间，服务器不能处理任何命令请求。
    • bgsave
      • bgsave命令会派生出一个子进程，然后由子进程复制创建RDB文件，主线程不会阻塞[服务器(主线程)继续处理请求命令]，从而避免主线程被阻塞。bgsave是Redis RDB文件生成的默认配置方式
      • 执行 bgsave 期间，服务器可以正常处理命令，但是 save ，bgsave，bgrewriteaof 三个命令与平时有所不同。
        • 在 bgsave 执行期间， 客户端再发送 save 命令时会被服务器拒绝。拒绝 save 和 bgsave 同时执行是为了 避免父进程(服务器进程)和子进程同时调用 rdbSave,防止产生竞态。
        • 在 bgsave 执行期间， 客户端再发送 bgsave 命令时，也会被拒绝， 因为执行两个 bgsave 命令也会产生竞争条件
        • bgrewriteaof 不能和 bgsave 命令同时执行 ：
          • 若正在执行 bgsave 命令，客户端发送 bgrewriteaof 命令，则会被延迟到 bgsave 命令执行完毕之后再执行。
          • 若正在执行 bgrewriteaof命令，客户端发送 bgsave 命令，则会被服务器拒绝。
+ **RDB文件的实际上由rdbSave()完成**，save和bgsave以不同的方式调用该函数。save 和 bgsave 的主要区别在于**会不会阻塞主线程**：save 是由服务器进程执行保存工作，bgsave是由子进程执行保存工作，前者会阻塞服务器，后者则不会。bgsave 命令执行全量快照，既提供了数据的可靠性保证，也避免了对Redis性能的影响。

  • 服务器在载入 RDB 文件期间，会一直处于阻塞状态，直到载入工作完成为止。

  • 对于不同类型的键值对，RDB文件会使用不同的方式来保存它们。

    RDB快照思考问题

  • 在执行快照期间能够修改数据吗？

    • 不能修改，但是可以通过写时复制技术来实现快照期间数据可修改
    • 为了能够在执行快照过程中，正常处理写操作，Redis借助了操写时复制技术(Copy-On-Write,COW)。
      • bgsave子进程由主线程fork生成的，（通过拷贝主线程的页表）可以共享主线程的所有内存数据，bgsave子进程运行时，读取主线程的内存数据，并把它们写入RDB文件。
      • 若主线程也是读操作，则主线程和 bgsave子进程互不影响。
      • 若主线程要修改某一块数据，则这块数据会被复制一份，生成该数据的副本。bgsave 子进程会把这个副本数据写入RDB文件，而主线程则可以直接修改原来的数据。
    • 通过写时复制(COW)技术既保证了快照RDB的完整性，也允许主线程对数据的修改。

  • 多久做一次快照？可以每秒做一次快照吗？

  • 频繁地执行全量快照，会带来两方面的开销

    • 频繁地将全量数据写入磁盘，会给磁盘带来很大的压力，多个快照竞争优先的磁盘带宽，前一个快照还未写完，后一个又开始了，导致恶行循环。
    • bgsave子进程需要通过fork操作从主线程创建处理，虽然bgsave子进程在创建后不会阻塞主线程，但是 fork创建过程本身是会阻塞主线程的，主线程内存越大，阻塞越久。若频繁fork出bgsave子进程，会频繁阻塞主线程。

  • 可以采用 增量快照

    • 增量快照指的是做了一次全量快照后，后续的快照只对修改的数据进行快照记录，从而避免每次全量快照的开销。
    • 增量快照的前提是：需要记住哪些数据被修改了。但记录这些元数据信息，又需要额外的空间开销。
    • 与AOF相比，快照RDB恢复速度快，但是频率不好把握。
      1. 频率太低，两次快照键一旦宕机，就可能丢失较多数据。
      2. 频率太高，会导致额外开销。

  • Redis 4.0提出了一个 混合使用AOF日志和内存快照 的方法

    1. 内存快照以一定的频率执行执行。避免频繁执行快照带来的开销。
    2. 在两次快照之间，使用AOF日志记录这期间的所有命令操作。AOF日志只记录两次快照间的操作，而不是所有操作，避免出现日志文件过大的情况，同时避免重写开销。

  • RDB内存快照的优势在于，可以快速恢复数据库，即只需要把RDB文件读入内存内存，避免AOF需要顺序、逐一执行每条命令带来的低效性问题。

RDB总结

  1. RDB文件用于保存和还原Redis数据库中所有的键值对
  2. save 命令由主线程直接执行保存操作，因而会阻塞服务器；bgsave 命令则单独创建一个子进程执行保证操作，所以不会阻塞服务器。
  3. RDB文件的实际上由rdbSave()完成**，save和bgsave以不同的方式调用该函数。save 和 bgsave 的主要区别在于会不会阻塞服务器
  4. bgsave是Redis RDB文件生成的默认配置方式，当条件满足任意一个设置的 save 条件时，则会自动执行 bgsave命令
  5. RDB 是一个经过压缩的二级制文件，由多个部分组成；不同类型的键值对，RDB文件会使用不同和的方式来保存。

高性能IO模型：为什么单线程Redis能那么快？

Redis单线程

  • 通常所说 Redis 的单线程，主要是指 Redis的网路IO和键值对的读写是由一个线程来完成的。但是Redis的其他功能比如持久化、异步删除、集群数据同步等，其实是用了额外线程来完成的，

    Redis为什么用单线程

  • 多线程 与 单线程
    • 在合理的资源分配情况下，多线程可以增加系统吞吐率。提升系统同时处理并发请求数量。
    • 但是多线程系统中通常存在某些临界资源,对这些临界资源的访问就需要互斥地访问，而为了保证互斥访问，就需要引入额外的机制来加入保障比如锁，但这也会增加额外的开销。
    • 并发访问控制是多线程编程开发的难点，如果没有精细的设计，比如只是简单采用粗粒度的互斥锁，往往会导致不理想的结果，即使增加了线程，大部分线程也在等待获取访问共享资源的互斥锁，导致并行变串行，使得系统吞吐率没有随着线程的增加而增加。
    • 多线程开发中一般都会引入 同步原语 来保护共享资源的并发访问， 但这会降低易调试性和可维护性。多线程编程模式面临的共享资源的并发访问控制问。
  • 为避免以上这些问题，Redis直接采用了单线程模式，单线程不会出现资源竞争的问题，从而也就不会有并发控制的开销问题。

    单线程 Redis 为什么那么快？

  1. Redis作为内存数据库，大部分操作在内存中完成，而内存操作本来就快。

  2. Redis优化设计的高效数据结构,比如sds, ziplist;Redis是一个KV数据库，是基于全局哈希表的，通过key值可以很快获取value值。

  3. Redis使用的是单线程，无切换开销。

  4. Redis采用的基于非阻塞的IO多路复用机制，使其在网路IO操作中能并发处理大量客户请求，实现高吞吐率。

    • 在网路IO操作中，早 accept() 和 recv()等操作中可能会阻塞

      • 比如 Redis监听某个连接请求，但是连接一直为成功，则会阻塞在accept，导致其他连接请求无法得到响应，
      • recv()类似，数据一直未到达，则Redis阻塞在recv()

    • 因此，Redis采用非阻塞IO模型：比如调用listen方法，将主动套接字转化为监听套接字，进行监听客户端连接请求。

      • 针对监听套接字，可以设置非阻塞模式，使得redis调用accept()方法但一直没有客户端连接到达时，redis线程可以返回干别的事，不用阻塞在监听这个地方[有数据达到时通知Redis]

Redis为什么又引入多线程？

  • 是在Redis6.0引入多线程的。
  • 引入多线程并非是抛弃了单线程。
    • 引入多线程是为了解决redis的网络IO瓶颈，使用多线程处理网络数据的读写和协议的解析,从而提升网络IO读写效率，同时可以利用服务器的多核资源。
    • 执行命令的还是单线程。

什么是热key ？ 热key的问题怎么解决

  • 所谓 热key问题就是，突然有大量的请求比如几十万去访问redis某个特定的key,那么这样会导致流量过于集中，而到达物理网卡的上限，从而导致这台redis服务器宕机引发雪崩。
  • 针对热key的解决方案:
    1. 提前把热key打散到不同的服务器,降低压力
    2. 加入二级缓存，提前加载热key数据到内存中，若redis宕机，走内存查询。
