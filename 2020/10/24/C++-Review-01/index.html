const与 #define区别

    • 处理的阶段不同

      • #define 是宏定义，在预处理阶段就进行文本替换。
      • const定义的是只读常量，在编译阶段才处理。

    • 类型安全不同

      • #define是宏定义，不会进行类型检查，只是简单的文本替换。
      • const定义的是常量，在编译阶段会进行类型检查。

    • 存储方式不同

      • #define 不分配内存，在替换是对变量分配内存，因而在内存中可能存在多份拷。
      • const常量在内存分配内存，存储在静态变量区，只存在一份。

    • 能否调试

      • #define 定义的常量不能调试。
      • const 定义的常量是可调试的。

引用和指针的区别

  • 指针和引用都能提供对其他对象的间接访问，但在具体细节却不同,最重要一点就是引用本身并非一个对象，只是一个别名而已
    • 指针有自己的一块空间，而引用只是一个别名。
    • 指针可以被初始化为NULL，而引用必须被初始化且必须是一个已有对象的引用。
    • 使用sizeof看一个指针的大小是4，而引用则是被引用对象的大小。
    • 指针在使用中可以指向其他对象，但是引用一旦绑定，则无法在更改绑定对象。

常量指针与指针常量

    • 常量指针本质是指针，常量修饰它，表示这个指针乃是一个指向常量的指[变量]，指针指向的对象是常量，那么这个对象不能被更改。
    • 指针常量的本质是常量，而用指针修饰它，指针常量，只能指向该对象，该对象是一个变量。
    • 常量指针常量 : 指针和指向的对象都不能变

函数指针与指针函数

  • 指针函数: 是返回值为指针的函数
  • 函数指针：是指向函数的指针,主要在回调函数时使用。
    • 函数指针与其他类型指针类似，只不过指向的是函数的地址。
    • 函数明在被使用时总是由编译器把它转为函数指针。

智能指针

原因: 管理动态内存是一种很棘手的问题，常规指针可能会存在忘记释放内存导致内存泄漏;尚有指针指向内存，但该内存却被释放了的非法内存的指针以及二次释放等等问题。

为了更容易(安全)使用动态内存，C++11中提供了两种智能指针(smart pointer)类型来管理动态对象。智能指针是行为类似常规指针的类对象，主要区别在于它可以自动释放所指向的对象，主要用到了RAII思想

C++11主要引入了三种智能指针，shared_ptr、weak_ptr和unique_ptr,它们的主要区别在于底层管理指针的方式
: shared_ptr 基于计数器技术、允许多个指针指向同一个对象; unique_ptr则是基于所有权模式,”独占”所指向的对象。C++11还引入了 weak_ptr 智能指针，它是一种弱引用，指向 shared_ptr对象，用于解决 shared_ptr循环引用的问题(因此，sharwd_ptr、weak_ptr归为一类);这三类智能指针都存在于 memory 头文件中。

智能指针特征

  • 智能指针(smart pointer)是模板类，是行为类似指针的类对象，用于生存期控制，能够确保自动正确的销毁动态分配的对象，防止内存泄露（利用自动调用类的析构函数来释放内存）。它是C++ RAII的一种应用。它可以使用使用引用计数(shared_ptr)和所有权模式(unique_ptr)两种方式。

  • 默认初始化的指针指针保存着一个空指针:

    shared_ptr

  • std::shared_ptr 是一种基于计数的智能指针，它能够记录多少个shared_ptr 共同指向一个对象，并维护一个共享的计数器。

  • 基于计数器的智能指针类将一个计数器与类指向的对象相关联，计数器跟踪该类有多少个对象共享同一指针。

    • shared_ptr 通常使用 make_shared()函数进行创建，创建时，该对象的引用计数置为 1
    • shared_ptr 计数器增加
      • 拷贝一个 share_ptr时
      • 赋值时，被赋值的对象
    • shared_ptr 计数器减少
      • shared_ptr被赋新值时
      • 离开作用域时

  • 一旦一个 shared_ptr 的计数器变为 0 , 它会自动释放管理的对象

  • shared_ptr原理上是基于计数的，但具体实现上是基于计数器或其他数据结构来记录有多少指针共享该对象，则由具体实现时决定。关键是能记录有多少个shared_ptr指向相同的对象，且在适当的时候自动释放对象。

  • shared_ptr 的析构函数会递减它指向的对象，若引用计数变为0，则shared_ptr的析构函数就会销毁对象，释放资源。

    weak_ptr

  • shared_ptr有一个问题即循环引用问题，若存在循环引用，会导致shared_ptr的引用计数不能变为0，从而不能释放资源。为了解决该问题，就引入了weak_ptr,weak_ptr是一种弱引用，用于配合shared_ptr的使用，弱引用不会引起引用计数的增加且并不影响对象的生命周期，也即是weak_ptr存在与否不影响对引用计数。

unique_ptr

  • 是一种独占的智能指针，基于所有权模式，任意时刻只能有一个拥有者，不支持拷贝构造和赋值， 但是可以利用 std::move() 将其转移给其他 unique_ptr。
  • C++14 中引入了 std::make_unique(),来创建 unique_ptr 指针

RAII(Resource Acquisition Is Initialization)资源获取即初始化
思想

    • RAII是一种利用对象生命周期来控制程序资源（如内存、网络连接等）的技术。
      • 用法：在对象构造时获取资源，使用资源，在对象析构时释放资源，因此，把管理资源的认为交给了一个对象。应用比如智能指针。
      • 好处：不需要显示的释放资源，在对象所需的资源在其生命周期内始终保持有效.
    • RAII的本质是用对象代表资源，把管理资源的认为交给对象,把资源的申请和释放与对象的构造和析构对应起来，从而确保在对象的生命周期内资源始终有效，在对象销毁时资源被释放，避免内存泄漏等情况的发生。也即使对现在，资源在，对象消失，资源释放。

智能指针是怎么实现的？什么时候改变引用计数？

  1. 构造函数中计数初始化为1；
  2. 拷贝构造函数中计数值加1；
  3. 赋值运算符中，左边的对象引用计数减一，右边的对象引用计数加一；
  4. 析构函数中引用计数减一；
  5. 在赋值运算符和析构函数中，如果减一后为0，则调用delete释放对象。

new 与 malloc的区别、原理等？

new和malloc区别

    • 类型不同

      • new/delete是c++里面引入的运算符。
      • malloc和free是标准库函数。

    • 申请内存的所在位置不同

      • new操作符是从自由存储器(free store)上为对象动态分配内存空间，自由存储区释放可以是堆(或是new是否能够在队上动态分配内存)，这取决于 operater new的实现细节。自由存储区不仅仅可以是堆，还可以是静态存储区。 new 甚至可以部位对象分配内存，而是使用已分配的内存，使用 定位 new 来做到。
      • malloc()则是从堆(heap)上分配内存。堆是操作系统中的术语，是操作系统所谓五的一块特殊内存，用于程序的动态内存分配。

    • 返回类型安全性

      • new操作成功是返回对象类型的指针，类型严格与对象匹配，无须进行类型转换。因此new是类型安全性的操作符
      • malloc内存分配成功返回void，则需要强制类型将void指针转为需要的类型。

    • 内存分配失败时的返回值

      • ew内存分配失败时，会抛出bac_alloc异常，而非NULL
      • malloc分配内存失败时返回NULL。

    • 是否需要指定内存大小

      • 使用 new 操作符申请分配内存时无须指定内存块大小，编译器会根据类型自行计算
      • malloc则需要显示指定所需内存的大小。

    • 对数组的处理

      • C++提供了new[]和delete[]来专门处理数组类型，必须配合使用。
      • malloc只是分配一块原始内存给你，不管该空间放置的东西，也不做初始化。

    • 是否可以重载

      • operator new/delete可以被重载
      • malloc/free不允许重载

有了malloc，为什么还需要new ？

    - 对于非内置的数据类型的对象而言，使用malloc/free是无法满足要求的，对象在创建时要自动执行构造函数，在对象消亡前要自动执行析构函数，而malloc/free是库函数，不在编译器的控制权限之内，不能够把执行构造函数和析构函数的任务强加给mallo/free。因此，C++需要一个能完成动态内存分配和初始化的运算符和清理与释放内存工作的运算符。
    + new操作符分配内存时会经历三个过程：
        - 第一步是调用 operator new函数，分配一块合适的，原始的，未命名的内存空间。
        - 第二步是编译器运行相应的构造函数以构造该对象。
        - 第三步是对象构造完成后，返回一个指向该对象的指针。

new、operator new 和 placement new的区别

  • new operator: 不能被重载，

      + new的执行过程如下：
      - 调用operator new 分配内存
      - 调用并执行构造函数
      - 返回相应类型的指针
  + delete过程 ：
      - 调用析构函数 
      - 调用operator delete,释放空间

    • operator new

      • 实现内存分配，重载时new时，实际上重载的是operator new,
      • 要实现不同的内存分配行为，应该重载operator new

    • placement new：

      • operator new 只是operator new的一个重载版本
      • 并不分配内存，只是返回指向已分配的某段内存的一个指针
      • 因此，不能删除，但需要调用对象的析构函数。
      • placement new 允许在一个已分配的内存中构造一个新对象。
      • 引入 placement new的理由：
        • 用 placement new 解决buffer问题：
          若是用new分配数组缓冲时，先要申请分配内存，然后调用构造函数，效率不急啊；使用 placement new 可以直接在已预分配的内存上构造函数。
        • 增大时空效率问题：
          使用 new 申请分配空间时，需要在堆中查找足够大的空间，然后分配，这个操作比较慢，同时可能出现内存不足，内存分配异常；而 placement new 是在一个预先已分配的内存缓冲区将那些，不需要查找内存，这样内存分配的时间就就是常数，且不会出现malloc内存不足的异常。因此，placement new非常适合对时间要求较高，长时间运行不希望被打断的应用程序。

malloc内存分配原理

    • malloc在申请内存时，一般会通过brk或者mmap系统调用进行申请空间。malloc申请内存小于128k时，使用 brk分配内存，申请内存大于128k时，使用mmap分配内存
    • brk和mmap这两种方式分配的都是虚拟内存，而非物理内存；当第一次访问已分配的虚拟地址空间时，会发生缺页中断，操作系统负责物理分配物理内存，然后建立虚拟内存和物理内存之间的映射关系。brk与sbrk， mmap与munmap
    • brk/sbrk/mmap属于系统调用，如果每次申请内存，都调用其中一个，系统调用开销较大，若每次都用系统调用，则会影响性能。
    • 这样申请内存容易产生内存碎片，因为堆是从低地址往高地址方向增长，若低地址没有释放，高地址内存就不能回收。
    • 因此，malloc采用内存池的方法
      ,来减少内存碎片的产生和减少系统调用的开销。先申请一个大内存，然后将其分为不同大小的内存块，并以块为管理的基本单位。用户申请内存时，就直接从内存池中选择一块相近的内存块即可。
    • malloc使用链表来管理内存块：
      • 隐式空闲链表
        :将堆区分为地址连续，大小不一的块，包含已分配和未分配的内存块。
      • 显示空闲链表
        :用一个链表将可用的内存块连接起来，组成一个双向的空闲链表，链表节点含有一个前向指针和后继指针，每个节点记录一个地址连续，未被使用的内存块，节点同时记录内存首地址和大小。

malloc之ptmlloc/tcmaclloc/jemalloc：

    • gcc默认使用的是ptmalloc分配器对内存碎片进行优化，但是这种优化不理想，胆汁看似内存泄漏，实际上时内存碎片的问题。
    • 为了更好解决该问题，google开发了tcmalloc和jemalloc内存管理工具，比如在Redis中就使用的时tcmalloc和jemalloc。
    • tcmalloc/jemlloc原理很类似，都是在链接时期替换标准库的maloc和free，在不改变diamond情况下，解决内存碎片的问题。
    • tcmalloc就是一个内存分配器，管理堆内存，主要影响malloc和free，用于降低频繁分配、释放内存造成的性能影响，且有效地控制内存碎片。
    • tcmalloc比ptmalloc快，一次malloc和free操作，ptmalloc需要大约300ns，而tcmalloc只需要50ns。+ + tcmalloc优化了存储对象，需要的空间更少，tcmalloc对多线程做了优化，对于小对象的分配基本不存在锁竞争

  • 内存泄漏定位:
    1. 在windows平台下通过CRT中的库函数进行检测；
    2. 在可能泄漏的调用前后生成块的快照，比较前后的状态，定位泄漏的位置

        • Valid-Value表：对于进程的整个地址空间的每一个字节(Byte）,都与之对应的 8 bit，这些bit负责记录该字节或寄存器的值是否有效，是否被初始化等。（CPU的每个继承寄存器，也有与之对的bit向量）
        • Valid-Address 表：对于进程整个地址空间中每一个字节（Byte）,还有与之对应的1bit,负责记录该地是否能够被读写。
      • MemCheck原理
        :
        • 当要读写某个字节时，首先检测该字节的 A bit,若A bit显示无效位置，memcheck则报告错误。
        • 当字节加载到cpu时，它对应的 v bit也被加载到cpu环境中，首先检测对应的 v bit，若为初始化，则memcheck报错。

虚函数及纯虚函数的理解

    • 虚函数: 只有成员函数可以为虚函数，在成员函数前面加上关键字 virtual

      • 虚函数作用： 实现多态
      • 实现原理
        :虚函数表（vtable）+ 虚表指针(vptr)
        • 虚表指针指
          向一个由虚函数的地址构成的一个数组（指针数组），含有虚函数的类有一个虚函数表，每个类对象由一个虚表指针，指向虚函数表
        • 虚函数表
          是存放着类中所有虚函数对象的函数指针的表，虚函数表类似一个数组，存储的是虚函数的地址，也即是虚函数的原始是指向类成员函数的指针。虚函数表的实现是编译器来实现的，编译器不同，可能实现方式不同

    • 虚函数表是属于类
      ,该类的所有对象共享这个虚函数表

    • 纯虚函数

    • 纯虚函数就是定义了一个虚函数但是没有实现，且在原型后面加了”=0”,含纯虚函数的类是抽象类，
      抽象类不能实例化对象，但是可以右指针和引用对象。

    • 构造函数可以是虚函数吗？

      • 不可以，原因如下：
      • 构造一个对象的时候，必须知道对象的实际类型，而虚函数是在运行期间确定实际类型的。但在构造一个对象时，由于对象还未构造成功，编译器无法知道对象的实际类型，比如是该类本身还是派生类，因为无法确定。
      • 虚函数的执行依赖于虚函数表，而虚函数表在构造函数中进行初始化，也即是初始化 虚函数指针vptr,让它指向虚函数表。但在构造对象期间，虚函数还没有被初始化。

    • 构造函数/析构函数中可以调用虚函数吗？

      • 从语法上来讲，这样调用是可以的。
      • 但是从效果上来看，往往达不到需要的效果[可以，但是没有意义，没有实现动态绑定的效果]

    • 正如 effective C++ 条款09 所讲： 绝对不要在构造函数或析构函数中调用虚函数。

    • 在继承体系中，构造的顺序 :基类构造函数 –>对象成员构造函数->派生类的构造函数析构的顺序刚好相反父类先于子类被构造，当父类的构造函数被调用时，子类还没有形成，此时子类对于编译器而是是不存在，所以调用的仍然是父类本身的虚函数。因为派生类对象构造期间进入基类的构造函数时，对象的类型变成了基类对象，而不是派生类对象，同样，进入基类析构函数时，对象也是基类类型。

    • 虚继承和虚基类

      • 虚继承
        是为了解决多重继承出现菱形继承的问题
        比如A是一个基类，B，C分别继承了A,D类多重继承B，C类， 则D中会出现两份A类数据。在进程时，通过在继承关系前加上 virtual 关键字即可实现虚继承。
      • 虚虚基类
        是用 virtual关键字声明继承的父类，即该基类在多条链路上被一个子类继承，但是该子类中只含一份该虚基类的数据。用来解决继承中的二义性问题
        。在同一层次中同时包含虚基类和非虚基类，那么先调用虚基类的构造函数，在调用非虚基类的构造函数，最后调用派生类的构造函数，析构则相反; 对于多个虚基类，则构造函数执行顺序从左到右。

左值和右值

    • 左值、右值的纯右值、将亡值、右值

      • 左值[lvalue:left-value]：顾名思义就是赋值符号左边的值。准确来说，左值是表达式（不一定是赋值表达式）后依然存在的持久对象
      • 右值[rvalue:right-value]: 右边的值，是指表达式结束后就不再存在的临时对象。
        • 纯右值[prvalue:pure-rvalue]: 字面量或匿名临时对象，非引用返回的临时变量、运算表达式产生的临时变量、原始字面量、Lambda 表达式都属于纯右值。
        • 将亡值][xvalue:expiring-value]:即将被销毁、却能够被移动的值，主要用于右值引用[T&&]。nullptr右值引用让临时变量的生命周期得以延长。
      • 左值与右值的主要区别： 左值可以取地址，右值不能;也即是左值是存储在内存中，而右值不是存储在内存中。

    • 左值是有名字的变量（对象），可以被赋值，可以在多条语句中使用。左值引用声明符号为 & 。

    • 右值是临时变量，没有名字，只在一条语句中出现，不能被赋值，右值声明符号为 &&。(左值：int &a = 1，右值：int &&a=1)

    • 凡是真正的存在内存当中，而不是寄存器当中的值就是左值，其余的都是右值。其实更通俗一点的说法就是:凡是取地址（&）操作可以成功的都是左值，其余都是右值。

    • 左值[lvalue : locator value]
      : 存储在内存，有明确地址的值

    • 右值[rvalue : read value]：
      提供数据的值，不一定可寻址；比如寄存器中的数据。右值又可以分为纯右值和将亡值

    • 有名称的，可以获取存储地址的值即为左值，否则为右值

    • 左值引用 T& 右值引用 T&&c) 同

    • 左值引用T&一样，右值引用T&& 创建时就必须初始化且只能使用右值进行初始化
      左右值最大的区别就是是否可以取地址，左值可以，右值不可以。

    • 右值引用(T&&)主要是为了用于 移动语义和完美转发

右值引用的意义:移动语义和完美转发

  • 右值引用的引入用来解决 C++ 中历史遗留问题，消除注入std::vector,std::string之类的额外开销。也使得 std::function()成为可能。

    • 右值主要用于 移动语义 和 完美转发

    • 为临时变量延长生命周期
      。因为右值在表达式结束后就消亡了，如果想继续使用右值，那就会动用（昂贵的）拷贝构造函数。

    • 移动语义
      ：将资源从一个对象转移到一个对象，避免不必要的拷贝，主要是将所有权转出去，避免申请空间和数据复制来提高效率。

      • 传统C++ 通过拷贝构造函数和赋值操作符为类对象设计了拷贝/复制的概念，但为了实现对资源的移动操作，调用者必须使用先复制、再析构的方式。
      • 右值引用可以消除两个对象交互时的不必要的对象拷贝，节省运算存储资源，提高效率。
      • 右值引用是用来支持转移语义的。转移语义可以将资源（堆，系统对象等）从一个对象转移到另一个对象，这样能够减少不必要的临时对象的创建、拷贝和销毁，能够大幅度提高C++应用程序的性能。

    • 完美转发：函数模板可以将自己的参数 完美 的转发其内部调用的函数

      • 。所谓完美转发，就是为了让我们在传递参数的时候，保持原来的参数类型（左引用保持左引用，右引用保持右引用）。一般使 std::forward()来进行参数的转发(传递)
      • 完美的意思是 不仅准确的转发参数的值，而且还能保证转发参数的左、右值属于不变。
      • 内部调用时使用模板函数 forward()，forward()函数用于转发，依赖于类型T
        • std::forward() 和 std::move() 一样，没有做任何事情
          • std::move()单纯地将左值转为右值
          • std::forward()也只是单纯地将参数做了一个类型转换，从效果上来看： std::forward(v) 和 static_cast<T&&>(v) 完全一样。
        • std::forward()包含两个重载，内部调用static_cast。

    • std::move()
      C++11引入的， std::move()可以将左值参数无条件的转为右值，std::move()实际上是 static_cast<T&&>()的简单封装。std::move()并不会move，只是做了类型转化而已，左值的移动操作时在移动构造或移动赋值完成的。

emplace_back()和push_back()区别

    • emplace_back()和push_back()都是在容器尾部添加元素是操作，但是它们的效率是不同的。
    • 在c++11引入右值引用之前，push_back() 是先调用构造函数创建一个临时对象，然后调用拷贝构造函数将临时对象拷贝到容器中，最后释放临时对象，有一个申请空间，拷贝，和释放空间的过程, 导致临时变量申请资源的浪费。
    • 在引入右值引用之后，则是创建临时对象，然后使用移动构造函数，将里临时对象加入对象。从而避免了一次拷贝过程和开辟空间的消耗，在原来的基础上提高效率
    • emplace_back()则是在这基础上，进一步改进，直接在容器中就地构造，没有赋值和移动操作，进一步提高效率。

    类型推导 auto 和 decltype

  • auto

    • 来由: 有需要将表达式的值赋给变量，但是这需要表达式的类型，这一点做起来不容易，甚至完全做不到，为了解决这个问题，C++11新标准引入了auto。
    • auto 让编译器
      通过初始值来推算变量的类型，因此，auto 定义的变量必须有初始值
    • auto 一般用于变量类型，auto使用时就必须初始化，在使用迭代器的时候比较方便
    • auto类型推导的典型例子：就是迭代器
    • auto 推导出的类型有时候并不与完全和初始值一样，编译器会适当的改变类型使其更加符号初始化规则， auto 一般会忽略顶层的 const
    • auto 不能用于的地方：
      1. auto 不能用于函数传参
      2. auto 不能用于推导数组类型

  • decltype

    • 来由: 希望从表达式中推导出类型，但又不想用该表达式的值初始化变量(或执行该表达式)，为此，C++11引入了decltype
      。
    • 作用: decltype执行编译器去分析表达式类型，但不实际计算表达式的值[只获取表达式类型]。
    • decltype是为了解决 auto 关键字只能对变量进行类型推导的缺陷而出现的，用法与 typeof 类似
    • decltype用于表达式类型推导：在编译期间完成类型推导，只进行类型推导，不会实际执行表达式。
    • decltype(表达式)
    • decltype 和 引用：

      decltype((variable))[主要是双括号]的结果永远是引用，而 decltype(variable)结果只有当 variable 本身是引用时才是引用

  • 尾返回类型推导

    • 为了解决在运行后才能知道类型的问题，C++11引入了 尾返回类型推导[trailing-return-type],利用 auto 关键字将返回类型后置
      • C++14中可以直接让普通函数具备返回值推导，因此尾返回类型更加简单

    ++i是在原地操作，效率更高；i++选哟申请临时变量，++后再赋值回去，效率低(需要申请临时空间存储临时变量)

sizeof和strlen()的区别

    • sizeof 是一个操作符， strlen()是一个库函数
    • sizeof的参数可以是数据类，也可以是变量， 而strlen()只能以结尾’\0’的字符串作为参数
    • sizeof在编译时期就确定了结果，strlen()需要在运行时才能计算出结果
    • sizeof 计算的是分配的内存大小，strlen()则是字符串实际的长度
    • 数组作为sizeof参数时，不退化， 作为strlen()参数时，则退化为指针。

产生coredump文件的原因

    • 访问内存越界（如使用下标访问字符串未进行结束符判断，strcpy，strcmp,strcat等越界）

    • 非法指针(使用空指针或随意转使用指针转换)

    • 堆栈溢出

    • 多线程使用了线程不安全的函数

    • 多线程读写的数据没有加锁保护

    • 定位coredump文件：

      • 先编译生成含有调试信息的可执行文件 gcc –g test.cpp –o test
      • 在执行 ./test,从而产生coredump文件
      • 在执行 gdb ./test ./core
      • 在gdb下输入 where

缓存淘汰算法

    • LRU（least recently used, 最近最少使用），根据数据的历史访问记录来进行淘汰数据，其核心思想是“如果数据最近被访问过，那么将来被访问的几率也更高”。
      • 新数据插入到链表头部；
      • 每当缓存命中（即缓存数据被访问），则将数据移到链表头部；
      • 当链表满的时候，将链表尾部的数据丢弃。

进程通讯的几种方法

    • 管道。管道的机制类似于缓存，是单向传输的，效率较为低下，比如：a进程给b进程传输数据，只能等待b进程取了数据之后a进程才能返回。
    • 消息队列。优点：把进程的数据放在某内存以后，无需等待其他进程来取就返回。缺点：
      • 如果a进程发送的数据占内存比较大，并且两个进程之间通信较为频繁，消息队列模型就不合适，因为a发送的数据很大的话，发送消息（拷贝）这个过程需要花很多时间来读内存。
    • 共享内存。系统加载一个进程的时候，分配给进程的内存并不是实际物理内存，二是虚拟内存空间。将两个进程的虚拟内存空间映射到同一个实际物理内存中，就实现了内存共享的机制。
    • 信号量。为了解决共享内存的线程安全问题，引入信号量的通信方式。

编译的过程

    • 编译的分别四步, 也即是：预编译处理(.c) －> 编译、优化程序（.s）－>汇编程序(.obj、.o、.a、.ko) －> 链接程序（.exe、.elf、.axf等），具体如下：

    • 预处理 : 预处理器指令处理 # 号开头的，比如#include, #define;一般来讲因为引入了头文件，预处理编译的文件比源文件大，将 .cpp文件转为 .i 文件。

      • 命令 g++ -E test.cpp -o test.i
      • 主要处理以 # 开始的预编译指令， 如 #include、 #define 和 注释
      • #pragma 则保留，编译器需要它们

    • 编译 : 通过预处理后的 .i 文件中，只有常量，数字，字符串及关键字等，编译主要是通过语法分析和词法分析，来确定所有指令是否符号规则，并将其转换成汇编代码， 将 .i 文件转为 .s 文件。

      • 命令 ： g++ -S test.i –o test.s
      • 主要进行词法分析，语法分析，语义分析等
      • 汇编：将汇编语言翻译成目标机器指令的过程，将 .s 文件转为 .o 文件。
      • 命令 ：g++ -c test.s –o test.o

    • 链接：汇编生成的目标并不能立即执行，还需要通过链接，因为 某个文件可能调用另外一个源文件中的函数或常量，或是函数中调用了库函数；链接的主要工作就是将有关目标文件连接起来， 将 .o 文件转为可执行文件。

      • 静态链接 ： 在编译期间就确定了
        • 对库函数的链接是在编译期间完成的。 Libxxx.a
        • 动态链接 ： 在运行时再载入
      • 在程序运行时期才载入

g++与gcc的区别

    • 它们都可以编译.c和.cpp文件；后缀为 .c 的文件， gcc把它当作C程序， g++ 则将它当作 C++ 程序，对于 .cpp文件，gcc和g++都将其视为 c++程序。
    • 编译阶段，g++会调用gcc，对于cpp文件，两者等价，但gcc 命令不能自动和C++程序使用的库链接。因此，对于cpp文件，一般使用g++。
    • 对于 extern ”C” ，gcc/g++都将以C的方式进行编译。
