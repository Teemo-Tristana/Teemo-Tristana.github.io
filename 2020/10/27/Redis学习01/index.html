发表于 2020-10-27 更新于 2020-11-12

Redis基本数据结构类型

  • Redis有6种基本类型的数据结构，分别是字符串string、列表linklist、哈希(字典)hastable、跳跃表skiplist、压缩列表ziplist和整数列表intset。

字符串string

  • redis 没有直接使用 C 语言原生的字符串表示，而是使用自己实现了简单动态字符串(simple-dynamic-string)

  • SDS 相对于 C 原生字符串的好处：

    • 获取字符串长度的时间复杂度为 O(1)

      • C 字符串没有记录字符串的长度，想要获取长度就需要遍历字符串，时间复杂度就为O(N)
      • sds 中属性 len 记录了sds中buf的长度，时间复杂度是O(1)

    • 防止缓冲区溢出[buffer-overflow]

      • C 字符串中没有记录自身长度，而是通过空字符串 \0 来判断结束，若在字符串没有结束符，则可能导致缓冲区溢出。比如
      • sds中进行拼接时，比如 sdscat()会预先检查空间是否足够，不够的时会通过 sdsMakeRoomFor 进行扩展空间。

    • 减少因修改字符串带来的内存重分配次数

      • c 字符串修改字符串长度 N 次必然需要执行 N 次内存重分配
      • sds 对于内存操作使用 空间预分配和惰性释放两种策略，因此字符串对修改字符串长度 N 次最多需要执行 N 次内存重分配，
        • 预分配内存
          • 若 sds 中字符未使用空间(alloc-len)够用,直接返回
          • 若需要的空间大于未使用的空间
            • 若 总需要的空间(newlen=len+addlen) 小于 SDS_MAX_PREALLOC(1M),则直接分配 2倍 newlen。
            • 若 newlen > SDS_MAX_PREALLOC，则每次分配 1M(SDS_MAX_PREALLOC)空间
        • 惰性释放
          当 sds 字符串执行缩短操作时，并不立即使用释放空间，只是修改sds中len属性，到某个时刻才会真正的释放,比如截断函数sdstrim()，并没真正的释放空间，而是修改len属性和添加\0 若用户真正想释放掉空闲空间时，可以使用 sdsRemoveFreeSpace()。

    • 可以存放二进制数据

      • c 字符串 只能保存文本数据，字符串里面不能包含空字符，否则会被认为时字符串结尾长度。
      • sds 字符串可以保存文本以及图片、音频、视频、压缩文件这样的二进制数据，sds的API是二进制安全的。

链表 linklist

  • redis的链表用用于 发布与订阅、慢查询、监视器等功能
  • redis的链表是是一个双向无环链表，每个链表节点由一个listNode结构表示。
  • redis的链表用 list 表示， 带有表头节点指针、表尾节点指针、链表长度信息等。
  • 通过链表设置不同的类型特定函数，Redisd的链表可以用于保存各种不同类型的值[类似多态]。
    • 链表节点使用 void* 指针来保存节点值
    • list的dup，free,matchs三个属性为节点值设置类型的特定函数
    • 因此链表可以用于保存各种不同类型的值

哈希(字典)hastable

  • 用于保存 键值对(key-value)的抽象数据结构，值与键关联，键在字典中是唯一的
  • C 语言中没有该内置类型，因此Redis自己实现了字典，底层采用hash表。一个哈希表中可以有多个哈希表节点，而哈希表节点就保存了字典中的一个键值对。

  • Redis的哈希表使用链地址法(separate chaining)来解决键冲突，每个哈希节点都含有一个next指针，多个哈希节点可以用next指针构成一个单向链表。
    • dictEntry 节点组成的链表是单链表，没有指向性链表尾部的指针，出于速度上的考虑，出现冲突时，总是将新节点添加到链表头部(复杂度为O(1))
  • rehah: 哈希表保存的键值对逐步增加增加或减少，为了让哈希表的装载因子保持在合理范围，会在适当时刻进行扩展或收缩，这项工作可以有 rehash 操作来完成。
  • Redis默认有两个全局哈希表，把ht[0]的数据重新映射并拷贝到ht[1]中(涉及到大量数据的拷贝，若一次性拷贝则可能导致Redis线程拥塞),为避免此问题，Redis采用了渐进式哈希即rehash
    1. 为字典的 ht[1] 哈希表 进行分配空间，大小取决于 ht[0]中键值对数量（ht[0].used属性）
      • 扩展: ht[1]的大小是第一个大于等于 ht[0].used*2的 2^n。
      • 收缩: ht[1]的大小是第一个小于等于 ht[0].used/2的 2^n。
    2. 将保持在ht[0]中的所有键值对rehash到ht[1]: 重新计算ht[0]的哈希值和哈希索引，然后将键值对放置到哈希表ht[1]指定的位置中。
    3. 待ht[0]包含的键值对全部迁入ht[1]之后(ht[0]变为空的hash表)，释放ht[0],并将ht[1]设置为ht[0],并新将一个空hash表作为ht[1],为下一次hash做准备。
  • 在渐进式hash执行过程中,新添加的字典的键值对一律保持在ht[1],而ht[0]不在进行任何添加操作，从而保证ht[0]包含的键值对数量只会减少，随着rehash的操作的执行而最终变成空表。

跳跃表skiplist

  • 跳跃表(skiplist)是一种有序数据结构,是在链表基础上，增加了多级索引[通过在每个节点中维持多个指向其他节点的指针]，通过索引位置的跳转，实现数据的快速定位，平均复杂度O(logN),最坏O(N)，大多数情况下，跳表的效率可以与平衡二叉树媲美。
  • Redis中只在两个地方使用跳表:
    • 实现有序集合键
    • 在集群节点中用做内部数据结构

      压缩列表ziplist

  • 压缩列表是为了节省内存而开发的顺序性数据结构。类似一个数组，数组中每个元素对应保存一个数据，每个数据有三个字段 zlbytes， zltail， zllen，分别表示列表长度，列表尾，列表中entry个数；压缩列表在表尾还有一个zlend，表示列表结束。

整数列表intset

  • 用于保存整数值的集合抽象数据结构，不会出现重复元素，底层是用数组实现，

Redis数据类型

  • 基于上述6种基本类型，Redis封装了自己的5种数据对象，每个对象至少用到了一种基本的数据结构.

  • Redis有5种数据类型，分别是:字符串String、列表List、哈希表Hash、集合Set和有序集合SortSet

  • redis通过 encoding 属性来设置编码形式，从而提升灵活性和效率，基于不同的应用场景redis自动做出优化具体如下

    1. 字符串对象string: int整数、embstr编码的简单动态字符串、raw简单动态字符串
    2. 列表对象list: linklist、ziplist
    3. 哈希对象hash: hashtable、ziplist
    4. 集合对象set: hashtable、intset
    5. 有序集合对象zset: ziplist、skiplist

对象

Redis对象

  • Redis实现了实现了字节的特定6中数据结构： 简单动态字符串sds、双向链表linklist、跳表skiplist、哈希表hashtable、集合set和有序集合。但是Redis并未直接使用这些基本的数据结构，而是基于这些数据结构实现了一个对象系统，这个系统包含：字符串对象、列表对象、哈希对象、集合对象和有序集合对象。其中，每个对象至少包含了一种基本的数据结构 。

  • Redis在执行命令之前，根据对象的类型来判断对象是否可执行给定的命令。

  • 使用对象的好处：

    • 针对不同的应用场景为对象设置多种不同的数据结构实现，从而优化对象在不同场景下的使用效率。类似多态[void * ptr](void*)可以转为任意类型，在不同场景选择不同的数据结构。

    • Redis的对象系统实现了 基于引用计数机制的内存回收机制。

      • 多个键可以共享一个对象来节约内存
      • 当引用计数为0[不在使用某个对象]时，该对象所占内存将被释放。

    • Redis对象带有访问时间记录信息，该消息可以用于计算数据库键的空转时长。空转时长较大的键可能会被优先删除。

      • 空转时长： 当前时间 - 键的值对象最后被访问的时间[lru时间]，单位 秒(s)

对象类型与编码

  • Redis使用对象来表示键和值，也即使是一个键值对至少有个两个对象:一个键对象，一个值对象

    • 键对象总是一个 字符串对象
    • 值对象可以是字符串对象、列表对象、哈希对象、集合对象、和有序集合

  • 通过 encoding 属性来设置对象使用的编码,而非固定类型，可以极大提升Redisd的灵活性和效率，因为Redis可以根据不同的使用场景来选择设置不同的编码，从而优化对象在某一场景下的效率[类似多态]。

    • 比如：在列表对象元素较少时，Redis使用压缩列表[ziplist]作为列表对象的底层实现：
      • 压缩列表ziplist比双端列表linklist更节省内存，而且在元素较少时，内存中以连续块方式保存的压缩列比链表可以更快的载入缓存。
      • 随着元素越来越多，链表的优势就体现出来[适合保存大量元素]，对象就会将底层实现从压缩列表ziplist转向双端链表linklist

字符串对象

  • 字符串是Redis五种类型对象中唯一一种会被其他四种对象嵌套的对象。
  • 字符串对象编码方式： int、 embstr、 raw[sds]
    • 对象是整数且可以用 long 类型表示 ：将 void* 转换为 long， 将对象编码设置为 int
    • 对象是字符串且字符串长度小于等于(<=)39字节: 使用 sds(简单动态字符串)保存，将对象编码设置为 embstr
      • embstr编码保存短字符串的好处
        • embstr将创建字符串对象所需要的内存分配次数从raw编码的两次变为一次
        • 释放embstr编码的字符串只需调用一次内存释放函数，而raw则需要两次
        • embstr编码的字符串保存在一块连续的地址空间，可以更好地利用缓存带来的优势。
    • 对象是字符串且字符串长度大于(>)39字节: 使用 sds(简单动态字符串)保存，将对象编码设置为 raw
    • embstr 与 raw 编码的区别：
      • embstr编码是专门用于保存短字符串的一种优化编码方式，过程和 raw 类似。但是 embrstr 只会调用一次内存分配函数来分配一块连续空间
      • raw 编码会调用两次内存分配函数来分别创建 redisObejct 和 sdshrd 结构
  • int编码和embstr编码 转为 raw 编码
    • int编码的字符串在对象执行某些命令后，不在是整数值时，编码方式将从 int 变为 raw 类型。比如 append 命令[追加命令]
    • embstr编码字符串对象实际上是只读的，若对embstr编码的字符串对象执行修改命令，则县将编码方式将从embstr转为raw，然后才会执行修改命令。
  • 字符串命令：
    • set
    • get
    • setrange
    • getrange
    • append
    • incrbyfloat
    • incrby
    • decrby
    • strlen

列表对象

  • 列表对象的编码方式：ziplist、linklist

  • 列表使用 ziplist 作为编码方式的条件：

    • 列表对象保存的字符串长度小于64字节
    • 礼拜对象的元素个数小于512个

  • 若不能同时满住以上两个条件，则列表对象将使用linklist作为编码方式(或对象的编码方式转换[ziplist->linklist])。

  • 列表命令：

    • lpush
    • rpush
    • lpop
    • rpop
    • lindex
    • llen
    • linsert
    • lerm
    • ltrim
    • lset

哈希对象

  • 哈希对象的编码方式： ziplist、hashtable
  • 列表对象使用 ziplist 作为编码的条件：
    • 哈希对象的键值对的键和值的字符串长度都要小于64字节
    • 哈希对象白菜的键值对数量小于512个
  • 若不能同时满住以上两个条件，则哈希对象将使用hashtable作为编码方式(或对象的编码方式转换[ziplist->hashable])。

生成时间 或 过期时间

  • 过期时间：过期时间是一个UNIX时间戳， 当键的过期时间过后，服务器就会自动从数据库删除该键
  • 设置过期时间：
    • expire ： expire 单位为 秒s
    • pexpire: pexpire 单位为 毫秒ms
    • expireat: expireat timestamp的时间戳 秒s
    • pexpireat： pexpireat timestamp的时间戳 毫秒ms
    • expire、pexpire和 expireat内部都是使用 pexireat来实现的
  • 移除过期时间：persist 是 pexpireat的反操作，用于移除过期时间

过期删除策略：

  • 定时删除：在设置过期时间的同时，创建一个定时器，让定时器在键过期的时候立即对键执行删除操作[主动删除]

    • 对内存友好，对CPU不友好

  • 惰性删除：键的过期时间到了不管，每次空键空间获取键时，判断键释放过期，若过期，就删除该键，若没过期就返回该键。[被动删除]

    • 对内存不友好，对CPU友好，可能存在过期键永远不会被删除的情况，从而导致 内存泄漏[无用的垃圾数据占用内存]。(某斜数据在某个时间后就可能不在访问，比如日志，但是过期数据仍然在内存中，用户以为服务器自动删除这些数据，但实际仍然在存在)

  • 定期删除： 每隔一段时间，就对数据库进行一次检查，以便删除过期键。[主动删除]

    • 定时删除和惰性删除结合： 执行惰性删除策略 + 每隔一段时间定期删除
    • 定期删除的难点在于 删除操作的时长和频率

  • Redis实际采用的是惰性删除和定期删除：通过配合使用这两种删除策略，服务器可以很好地在合理使用CPU时间和避免浪费内存指尖取地平衡。
