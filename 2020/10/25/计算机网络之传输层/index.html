计算机网络之传输层

发表于 2020-10-25 更新于 2020-10-26

原子操作

应用层

  • 作用：负责应用程序的逻辑，在用户空间实现

  • 常用协议

    • HTTP协议

      • 概念：Hypertext Transfer Peotocol 超文本传输协议 HTTP

      • 作用：规定浏览器与服务器之间通信的规则

      • HTTP是一种协议，只要通信双方都遵循这个协议，就可以使用http进行通信

      • HTTP1.1之前默认短连接，HTTP1.1及其之后默认长连接

      • HTTP和HTTPS

        • http和https使用的是完全不同的连接方式，用的端口也不一样，http端口是80，https端口是443

        • http是超文本传输协议，信息是明文传输，https则是具有安全性的ssl加密传输协议

        • HTTP默认端口是80， HTTPS默认端口是 443

        • HTTP是基于TCP的协议，HTTPS是在http和TLS/SSL结合的协议

        • https是针对http不足做出改进的

          • 窃听风险 -> 对传输的信息加密， 使用混合加密
          • 篡改风险 -> 采用完整性校验，使用数字签名
          • 冒充风险 -> 采用身份验证， 使用数字证数

      • HTTP组成

        • 判断 HTTP头部结束的依据是遇到一个 空行\r\n

          • HTTP协议是基于行的协议，每一行以回车换行符(\r\n)作为分隔符.空行及其之前的部分是头部，消息实体是body部分

        • 请求报文

          • 请求行
          • 首部行
          • 空行
          • 消息实体body（可选）

        • 响应报文

          • 响应行
          • 首部行
          • 空行
          • 消息实体body（可选）

      • HTTPS

        • 加密算法

          • 对称加密

            • 加密的密钥和解密的密钥是相同的，如AES和TEA，计算量小，但存在破解的风险。

          • 非对称加密

            • 加密和解密的密钥不同，公钥和私钥成对出现，公钥用于加密，私钥用于解密。计算量大，安全性强，常用的如RSA，ECC。

        • HTTP + SSL/TLS

      • HTTPS四次握手

        • SSL/TLS的四次握手

          • 四次握手过程

            • 1.客户端发出请求ClientHello ->

              • 客户端首先向服务器Client Hello消息，消息包含

                • SSL协议版本号，比如TLS1.0
                • 加密算法种类，以及其所需信息，如压缩方法
                • 包含客户端的一个随机数，用于生成对话密钥

              1. 服务器响应ServerHello <-

                • 服务器然后响应

                  • 验证客户端的证书
                  • 发送 Server Hello消息包含协议版本，加密方式等和服务端生成的一个随机数
                  • 发送服务器端证书
                  • 发送server key exchange消息，包含公钥和签名消息

            • 3.客户端再回应 ->

              • 发送 client key exhange，包含一个随机数，即预主密钥
              • 发送 change cipher spec消息，通知服务器端之后的数据段会加密传输
              • 发送finished消息，包含加密后的握手信息

            • 4.服务器响应 <-

              • 使用私钥将信息解密取出密码，再使用密码对消息解密并验证。之后发送消息时都会使用密钥进行加密，然后发送
              • 发送 chang cipher spec消息，通知客户端之后的数据段会加密传输
              • 发送finished消息，验证客户端的Finished消息并完成TLS握手

          • 注意事项

            • 3个随机数

              • client 会发送2个
              • server会发送1个
              • 作用：用于生成密钥

        • HTTPS整个过程需要完成7次握手： SSL四次握手 + TCP三次握手

        • HTTP/3 基于UDP的QUIC协议进行握手，将TCP和TLS的握手过程组合起来，从而只需要3次握手就可以建立安全的连接。

          • QUIC

            • 概念： quick udp internet connetion 快速UDP互联网连接（google提出的使用udp进行多路并发传输的协议）

            • 相比于http+tcp+TLS的优势

              • 减少了TCP三次握手及TLS的握手时间
              • 改进了拥塞控制
              • 避队头阻塞的多路复用
              • 连接迁移
              • 前向冗余纠错

      • 请求方法,方法名称区分大小写

        • get 与 post的区别

          • get方法不会修改服务器上的资源，它的等幂操作，而post是非等幂操作，可能会修改服务器的资源
          • get可以保存为书签，可以用来缓存，而post不可以
          • get把请求附在url上，而post把请求放在body里
          • get的长度一般有限制，不同的浏览器可能不同，但都在2k左右，post一般来说没有限制
          • get参数只支持ASCII，post的参数要广一些，比如可以用来传输二进制编码的信息

        • header

        • put

        • delete

        • trace

        • connect

        • patch

        • options

    • telnet 远程登录协议

    • OSPF(Open Shortest Path Furst 开放最短路径优先)

      • 动态路由更新协议
      • 用于路由器之间通信，告知对方各自的路由消息

    • DNS(Domain Name Service) 域名服务

      • 将域名转换到IP地址

  • ping ：是一个应用程序，不是协议

    • 用于检测网络是否联通
    • 利用ICMP报文检测网络连接情况

传输层

概述

  • 传输层的作用
    • 提传输层提供进程和进程间的逻辑通信， 网络层提供主机之间的逻辑通信
      • 分用和复用
    • 传输层主要有个两个协议： TCP和UDP
    • 寻址与端口：
      • 端口号 16 bit , 端口用来区分上层应用协议端口号是本地唯一标识用户进程
        • 共 2^16 - 1 = 65535 个 端口，是逻辑上的端口号
        • 知名端口: 0-1023
          • FTP 21
          • TELNET 23
          • SMTP 25
          • DNS 53
          • HTTP 80
          • HTTPS 443
        • 临时端口: 1024~65535
    • 套接字:
      • socket = 主机IP地址 + 端口号
      • 套接字socket是唯一标识网络中的主机进程

UDP协议(User Datagram Protocol)

  • UDP特点

    • UDP是基于数据报的不可靠的无连接服务

      • 数据报:

        • 每个UDP数据报都有一定长度，接收端必须以该长度为最小段位将所有内容一次性读出，否则数据将被截断
        • 发送和接受的数据报的次数是相同的
        • 不可靠
          • 尽量最努力交付“：不保证可靠交付
          • UDP无法保证数据从发送端正确地传输到目的端
        • 无连接:
          • 通信双方不建立连接，因此，每次发送数据时都需要明确指定接收端的地址字

      • 因此，使用UDP协议的应用程序需要自己处理数据确认，超时重传等机制。这些工作应该有UDP上层来完成

      • 面向报文：适合一次传输少量的网络应用

      • 无拥塞控制：适合很多实时应用：如IP电话，视频会议

    • UDP 首部格式

      • UDP首部只有 8B, 开销小

    • UDP校验

    • 无连接的UDP适合广播和多播

TCP：TCP是端到端的面向连接协议

  • TCP特点

    • 基于面向连接的可靠的字节流服务

      • 面向连接

        • 通信双方维持一个连接
        • 通信双方必须建立连接，然后才能进行数据传输
        • 通信双方必须为该挎包护额分配资源比如发送缓冲区和接收缓冲区，以管理连接的状态和数据传输

      • 可靠传输

        • 校验和
          • 校验释放是有效数据
          • 利用CRC算法，头部和数据部分都需要校验
        • 序号seq
          • 为每个字节都编号，用于解决乱序问题
          • RFC793规则：每次当一个新连接建立的时候，初始序列号生成器会生成一个32位的ISN(initial sequence number),范围在 0 ~ 2^32-1之间，绑定到一个32位时钟，低阶位大约每4微秒增加一次，因此ISN大于在4.5小时循环一次。而一个包在网络中生成时间都不会大于MSL,因此可以认为ISN是唯一的。
        • seq+1，因为SYN会占用一个序号
        • 滑动窗口
          • 滑动窗口用于解决流量控制问题
        • 重传
          • 超时重传:发送报文段后，立即计时，若时间到期仍然为收到确认，就重传该报文
          • 冗余重传：连续三次收到相同的ack时，就立即该报文段重传[在快重传中用到]
        • 确定机制ack
          • 对收到的报文段进行确认
          • 用于解决丢包问题

      • 字节流

        • 发送端执行的写操作次数和接收端执行的读操作之间没有任何对应关系
        • 数据的发送和接收是没有任何边界限制的
        • 类似 水流 一样
        • 字节流可能导致TCP粘包问题
          • 原因：粘包问题是由TCP是字节流协议，没有消息边界所引起的
          • 解决思路：解决粘包的方法应该由应用层进行分包处理，本质上由应用层来维护消息和消息的边界
            • 方法1：以特殊字符作为消息的分节符
            • 方法2：在消息头部加入消息的长度信息

    • TCP是点多点通信

  • TCP首部格式:

    • TCP首部至少 20 字节， 至多 60 字节

    • TCP报文段和UDP数据报都是通过其头部中的 16位端口号来区分上层应用程序

      • 头部字段
      • 16位 源端口号
      • 16位 目的端口号

      • 32位 序列号seq sequence number

        • TCP报文段传输过程中，会为每一个字节编号
        • 序号的初始值是随机值，后续的TCP报文段加上该随机值

      • 32为 确认号ack acknowledge number

        • 对接收到的TCP报文段的响应
        • 在收到的TCP报文段序号值基础上+1，表示下一次期待接收的序号

      • 4位 首部长度

        • 标识 该TCP头部有多少给32bit（4字节）
        • 4位最大值为15=16-1，因此TCP头部最长是60Byte

      • 6位 保留位

      • 6位 标志位

        • URG 表示紧急指针是否有效
        • ACK
          • 表示确认号是否有效
          • 携带ACK标识的TCP报文段称位确认报文段
          • PSH 提示接收端立即从TCP缓冲区读取该数据
        • RST
          • 表示要求对方重新建立连接
          • 携带RST的TCP报文段称位复位报文段
        • SYN
          • 请求建立连接
          • 携带SYN的TCP报文段称位同步报文段
        • FIN
          • 通知对方本段要关闭连接
          • 携带FIN的TCP报文段称位结束报文段

      • 16位 窗口

        • 窗口是TCP流量控制的一种方式
        • 这里的16位窗口表示接收窗口Receiver Window RWND
        • 告知对方本段TCP的接收缓冲区还能容纳多少字节的数据，从而调节对方发送速度

      • 16位 校验和

        • 由发送端填充，接收端对TCP报文段执行CRC算法，以检验传输过程种是否由损坏
        • 头部和数据部分都需要检验
        • 是TCP可靠传输的一部分

      • 16位 紧急指针

  • TCP连接管理

    • TCP通信双方必须建立TCP连接，并在内核中为该连接分配一定的资源

        -比如连接状态管理，读写缓冲区，定时器等

    • TCP三次握手

      • 所谓TCP三次握手是指建立一个TCP连接时，客户端和服务器需要总共发出3个包，三次握手的目的是连接服务器指定端口，建立TCP连接，并同步连接算法的序列号和确认，交换TCP窗口大小等。在socket编程中，客户端执行**connect()**时，将触发三次握手。
      • 三次握手的过程：
        • 第一次握手: 客户端发送一个 TCP 的 SYN 标志位置1的包，指明客户端打算连接的服务器的端口，以及初始序号 X,保存在包头的序列号(Sequence Number)字段里[SYN=1,seq=x],发送完后，客户端进入 SYN_SEND状态
        • 第二次握手: 服务器发回确认包(ACK)应答。即 SYN 标志位和 ACK 标志位均为1。服务器端选择自己 ISN 序列号，放到 Seq 域里，同时将确认序号(Acknowledgement Number)设置为客户的 ISN 加1，即X+1[SYN=1,ACK=1,seq=y,ack=x+1]。发送完毕后，服务器端进入 SYN_RECV状态
        • 第三次握手：客户端再次发送确认包(ACK)，SYN 标志位为0，ACK 标志位为1，并且把服务器发来 ACK 的序号字段+1，放在确定字段中发送给对方，并且在数据段放写ISN的+1[ACK=1，ACKnum=y+1]。发送完毕后， 客户端进入ESTABLISHED状态，服务器收到该包后，也进入ESTABLISHED状态
      • 一般都是clien通过**connect()**系统调用，主动与服务器建立连接
        • connect()失败原因
          • connect连接的端口不存在(未被任何进程监听)
          • connect连接的端口仍然被TIME_WAIT状态的连接所占用
          • 端口存在，但connect()连接超时
        • 连接在connect()失败后立即返回CLOSED状态
      • SYN烘泛攻击
        • 半连接: 在client处于SYN_SEND,服务器端处于SYN_RECV这样的状态,也即在TCP二次握手后三次握手前
        • 概念：
          • TCP处于半连接时：client端发送syn包请求连接，服务器为其分配系统资源，并回复确认包并等待对方确认的状态这种半连接状态会消耗系统资源
          • SYN洪泛攻击就是利用 半连接状态消耗系统资源的特性
            在短时间内伪造大量不存在的ip发送syn包，从而占用未连接队列，使得正常的SYN包别丢弃导致服务器资源消耗殆尽，严重时导致系统瘫痪。
        • 解决方法
          • 增加半连接数量
          • 缩短超时时间
          • SYN-cookies技术
          • 服务器收到SYN包并返回ACK和AYN时，不分配专门的数据区，而实根据收到的SYN计算出一个cookie值。

    • TCP四次挥手

      • TCP是全双工的，因此发送方和接收方都需要FIN和ACK,四次挥手细看实则是两次

      • 半连接状态：

        • TCP是全双工，因此它允许两个方向的数据传输被独立关闭
        • 一个方向连接被关闭，另一个方向仍然可以传输数据的状态比如FIN_WAIT_2状态

        • shutdown()是真正的关闭，可以指定关闭选项

          • SHUT_RD 关闭读端

            • 不能再执行读操作，若再对该套接字read则直接返回EOF
            • 读缓冲区已有的数据被全部丢弃
            • 如果有新数据到达，会对数据进行ack，然后悄悄的丢弃

          • SHUT_WT 关闭写端

            • 不能再对该fd进行写操作，若再进行写操作，则会触发SIGPIPE信号
            • 发送缓冲区已有的数据将被全部发送出去，最后在发送一个 FIN报文 给对端。
            • 关闭写端方向，常称为半关闭。

          • SHUT_RDWT 读写都关闭

            • 相当于 SHUT_RD和SHUT_WR操作各一次
            • 关闭套接字的read和write方向

        • 可能导致的问题: 孤儿连接

          • 孤儿连接：处于半关闭状态时，client未等服务器关闭连接就强行退出，导致该处于半连接状态的连接成为孤儿连接
          • font color=”#dd0000”>孤儿连接由内核来接管
          • 解决方法:
            • 未防止孤儿连接长时存在内中，有2个内核变量来控制孤儿连接到数量和生存时间
            • tcp_max_orphans : 内核中最大的孤儿连接数目
            • tcp_fin_timeout:孤儿连接的生存时间

      • MSL(Maximum Segment Life 报文段最大生存时间)

        • MSL是TCP报文段在网络中的最大生存时间
        • 2min或60s

      • TIME_WAIT状态为什么等待2MSL

        • 可靠地终止TCP连接, 保证第三次挥手被正确的接收
        • 保证让迟到的TCP报文段有足够的时间被识别并丢弃

      • 判断对方是否关闭连接的方法

        1. read()系统调用读取时返回 0(表示收到结束报文)

    • TCP状态转移

    • TCP流量控制：利用滑动窗口实现流量控制wnd = min(rwnd, cwnd)

      • 发送窗口SWND
      • 接收窗口RWND
        • TCP为每个连接设置有一个计时器，若接受窗口大小为零时，发送方停止发送，计时器开始计时，若计时器到期，还未收到对方消息。
        • 发送方就会发送一个探测报文给对端，大小为1字节接收方收到探测报文后，会就立即告知本端的接受窗口大小。接受窗口又可用时，会告知发送方可以继续发送.
      • 拥塞窗口CWND

    • TCP拥塞控制算法

      • 作用： 提高网络利用率，降低丢包率，保证网络资源对每条数据流的公平
      • 慢开始 & 拥塞避免
        • 慢开始：从报文段1开始发送，指数增长直到到达门限
        • 拥塞避免： 从门限值开始线性增加，每次增加1个MSS直到出现网络拥塞
      • 快重传 & 快恢复
        • 快重传:
          • 当发送方连续收到三个相同的ack时，就认为该报文段丢丢失，立即重传而不必继续等待设置的重传计时器时间到期。
          • 发送方只要一连收到3 个重复确认就应当立即重传对方尚未收到的报文段，而不必继续等待设置的重传计时器时间到期。
          • 因为一旦出现丢包，那么立即减半退避
        • 快恢复:当网络出现拥塞时，将拥塞窗口设置为出现拥塞时窗口的一半，再执行拥塞避免算法；从而实现cwnd快速恢复。

    • 流量控制和拥塞控区别

      • 拥塞控制主要是为了防止过多的数据注入到网络，导致网络拥塞；
      • 流量控制是防止方法一下子发送过多的数据到接收方，导致接收方缓存不下。
      • 两者都是对发送方行为进行控制。但拥塞控制是一个全局问题，流量控制是一个对点问题
      • 拥塞控制是考虑网络的拥塞情况，流量控制是考虑从发送速率与接收速率的情况

    • Nagle算法

      • 目的：减轻网络负载
      • 概念: Nagle 算法的改进在于：如果发送端欲多次发送包含少量字符的数据包，则发送端会先将第一个小包发送出去，而将后面到达的少量字符数据都缓存起来而不立即发送，直到收到接收端对前一个数据包报文段的ACK 确认为止，或当前字符属于紧急数据，或者积攒到了一定数量的数据等多种情况才将其组成一个较大的数据包发送出去。
      • Nagle算法主要是避免发送小的数据包，要求TCP连接上最多只能有一个未被确认的小分组，在该分组的确认到达之前不能发送其他的小分组。

  • TCP：TCP是端到端的面向连接协议

    所谓的‘端到端”可以理解为进程到进程之间的通信，

    • TCP特点

      • 基于面向连接的可靠的字节流服务

        • 面向连接

          • 通信双方维持一个连接状态
          • 通信双方必须先建立连接，然后进行数据传输
          • 通信双方必须位该连接分配必须的内核资源，以管理连接的状态和连接上的数据传输

        • 可靠传输

          • TCP协议采用发送应答机制

            • 发送的每个TCP报文段必须得到接收方的应答，才认为该TCP报文段传输成功
            • 发生buffer已发送的数据只有在收到确认后才会丢弃

          • TCP协议采用超时重传机制

            • 发送端每发送一个TCP报文段之后会启动定时器，若在规定时间那边未收到应答，则重复该报文

          • TCP协议对收到的TCP报文段重排，整理等处理之后在交付应用层

          • TCP可靠的具体体现

            • 利用校验和

              • 保证数据在传输过程没出现差错
              • 发送报文段的二进制相加然后取反，从而检测数据在传输过程中是否有变化。
              • 若校验和有差错，TCP丢弃该报文段且不确认该报文段。

            • 应答机制+序列号

              • 为每一个字节编号，接收方对数据进行排序
              • 对收到的报文段进行确认

            • 超时重传机制

              • 当TCP发送出一个报文段时同时启动一个定时器
              • 若超过预定实际还未收到对该报文段的确认，就认为该报文段丢失，重发这个报文段

            • 流量控制

              • TCP连接的每一方都有接收缓冲区。接收方每次对报文确认时会在报文头里面附带本端接收窗口的大小
              • 发送方选取接收窗口和拥塞窗口中最小的一个为发送窗口大小，进而实现调整发送速率
              • 利用滑动窗实现流量控制，动态改变发送窗口大小

            • 拥塞控制

              • 根据网络的拥塞情况动态调整发送速率
              • 策略：慢开始，拥塞避免和快重传及快恢复

        • 字节流

          • 发送端执行的写操作次数和接收端执行的读操作之间没有任何对应关系
          • 数据的发送和接收是没有任何边界限制的

            • SYN-cookies技术

              SYN Cookie 是对 TCP 服务器端的三次握手做一些修改，专门用来防范 SYN Flood 攻击的一种手段。
              原理是，在 TCP 服务器接收到 TCP SYN 包并返回 TCP SYN+ACK 包时，不分配一个专门的数据区，而是根据这个 SYN 包计算出一个 cookie 值。这个 cookie作为将要返回的 SYN ACK 包的初始序列号。
              当客户端返回一个 ACK 包时，根据包头信息计算 cookie，与返回的确认序列号（初始序列号 + 1）进行对比，如果相同，则是一个正常连接，然后，分配资源，建立连接。

              • 服务器收到SYN包并返回ACK和AYN时，不分配专门的数据区，而实根据收到的SYN计算出一个cookie值。

      • 接收窗口RWND

    • TCP拥塞控制算法：

      • 作用： 提高网络利用率，降低丢包率，保证网络资源对每条数据流的公平

      • 慢开始 & 拥塞避免

        • 慢开始

          从报文段1开始发送，指数增长直到到达门限

        • 拥塞避免

          从门限值开始线性增加，每次增加1个MSS直到出现网络拥塞

      • 快重传 & 快恢复

        • 快重传

          当发送方连续收到三个相同的ack时，就认为该报文段丢丢失，立即重传而不必继续等待设置的重传计时器时间到期。

          发送方只要一连收到3 个重复确认就应当立即重传对方尚未收到的报文段，而不必继续等待设置的重传计时器时间到期。

          因为一旦出现丢包，那么立即减半退避

        • 快恢复

          当网络出现拥塞时，将拥塞窗口设置为出现拥塞时窗口的一半，再执行拥塞避免算法；从而实现cwnd快速恢复。

    • 流量控制和拥塞控区别

      • 拥塞控制主要是为了防止过多的数据注入到网络，导致网络拥塞；
      • 流量控制是防止方法一下子发送过多的数据到接收方，导致接收方缓存不下。
      • 两者都是对发送方行为进行控制。但拥塞控制是一个全局问题，流量控制是一个对点问题
      • 拥塞控制是考虑网络的拥塞情况，流量控制是考虑从发送速率与接收速率的情况

    • Nagle算法

      • 目的：减轻网络负载

      • 概念

        • Nagle 算法的改进在于：如果发送端欲多次发送包含少量字符的数据包，则发送端会先将第一个小包发送出去，而将后面到达的少量字符数据都缓存起来而不立即发送，直到收到接收端对前一个数据包报文段的ACK 确认为止，或当前字符属于紧急数据，或者积攒到了一定数量的数据等多种情况才将其组成一个较大的数据包发送出去。
        • Nagle算法主要是避免发送小的数据包，要求TCP连接上最多只能有一个未被确认的小分组，在该分组的确认到达之前不能发送其他的小分组。

    • TCP粘包问题

      • 粘包问题是由TCP是字节流协议，没有消息边界所引起的

      • 解决粘包的方法应该由应用层进行分包处理，本质上由应用层来维护消息和消息的边界

        • 方法1：以特殊字符作为消息的分节符
        • 方法2：在消息头部加入消息的长度信息

          • 过TTL后，报文将被丢弃，并返回一个ICMP报文

        • 收到的IP数据报不正确

          • 将其丢弃，并返回一个ICMP错误消息

  • IP数据报

    • IP头部：至少长20B，至多60B

      • 4位 版本

        • 4位 用来 指定IP协议的版本

      • 4位 首部长度

        • 用来标识该IP头部有多少个32bit(4字节）
        • 4位， 最大能表示 15 位， 因此IP头部最长是 4 *15 = 60 Byte
        • 单位是 4Byte

      • 8位 服务类型

      • 16位总长度

        • 整个IP数据报的长度
        • 单位是 1Byte
        • IP数据报最大长度 2^16 - 1 = 65535 个字节

      • 16位 标识

        • 用来唯一标识主机发送的每一个数据报

          • 初始值有系统随机生成

        • 标识报文段是否是同一家的

        • 同一个数据报的分片具有相同的标识

      • 3位 标志

        • 第一位：保留位

        • 第二位DF(Don’t Fragment)

          • 1 禁止分片

            • 若传输的数据大于MTU,且DF=1，那么该报文将被丢弃，并返回一个ICMP差错报文

          • 0 允许分片

        • 第三位MF(More Fragment)

          • 1 表示后面还有分片
          • 0 表示是最后一个分片

      • 13位 片偏移

        • 分片相对原始IP数据报的偏移
        • 除最后一个IP分片外，每个IP分片的数据部分的长度必须是8的整数倍
        • 片偏移单位是 8B

      • 8位 生成时间 TTL

        • TTL值可以防止数据报陷入路由循环
        • 数据报达到目的端之前允许的最大路由跳数，8位， 最大值255
        • 每经过一个路由器，TTL就-1，当TTL=0时，路由器将丢弃该数据报，并发送一个ICMP差错报文

      • 8位 协议

        • 8位协议用来区分上层协议
        • ICMP:1, TCP:6, UDP：17，

      • 16位 首部校验和

        • 由发送端填充，接收端使用CRC算法检验
        • 只检验 IP数据报头部 在传输过程中是否有损坏

      • 32位 源地址

      • 32位 目的地址

    • IP报文头中单位

      • 首部的总长度单位是 1Byte
      • 首部长度单位是 4Byte
      • 分片的偏移单位是8Byte

          • 240-255

    • 子网划分

      • 三级IP地址： 网络号 + 子网号 + 主机号

        • 三级IP地址： 网络号 + 子网号 + 主机号
        • 子网俺码： 主机号为0，其他为1

    • 超网：无分类编址CIDR

      • IP地址： 网路前缀 + 主机号

        • 格式： 128.14.32.0/20 ： 表示前20位是网路前缀

        • 地址俺码： 网络前缀全位1，主机位全为0

        • 将多个子网合并为一个较大的子网，构成超网【路由聚合】

          • 将网络前缀缩短
          • 最长前缀匹配

        • CIDR中子网可以全0或全1
